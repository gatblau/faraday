# 07 — Cross-cutting Concerns & NFRs

## Cross-cutting Concerns

| Concern | Stance |
|---|---|
| Authentication | **Applies** — OIDC sign-in via the daemon-owned auth UI for the user identity; the OS keychain for service tokens; corporate-API access via the backend `obo-broker` service (provider-pluggable; no default IdP). Sensitive (write) capabilities may require **policy-configurable step-up authentication / MFA** (OQ-1). Step-up is **challenge-driven (ADR-015)**: on a `401` RFC 9470 `insufficient_user_authentication` challenge from `obo-broker`, the daemon performs an IdP step-up via its consent UI (requesting the challenged `acr`) and retries once; the assurance rides only in the `id_token` `acr` claim, never a caller-supplied field (server counterpart: `../obo-broker/` ADR-014). The agent itself is not authenticated; it is treated as untrusted. |
| Authorisation | **Applies** — the Capability Manifest is the authorisation policy; per-session user consent gates first use of each capability; admin policy can pre-approve or pre-deny. Enforced controls are host + canonical path + method. |
| Logging | **Applies** — append-only audit log per outbound call (sizes, not bodies); stdout/stderr passed through a redaction filter before return. SIEM/OTLP export is **mandatory in real-credential mode (ADR-016)**, checked fail-closed at startup (no reachable sink ⇒ mock / non-sensitive-credentials-only); the exported stream is the authoritative record and the local `.jsonl` log is explicitly non-authoritative (user-writable). |
| Metrics | **Project default (ADR-027)** — request/response sizes and `durationMs` are captured in audit entries; **no separate OTel/metrics pipeline** for the dev-machine profile (a full OTel metric set is optional/off-by-default for fleets — FU-028). |
| Tracing | **Applies (ADR-027)** — `runId` correlates all audit entries + structured logs for a run; `agentId` recorded. No distributed OTel tracing for the dev-machine profile (optional for fleets — FU-028). |
| Rate limiting | **Applies** — `maxCallsPerRun` (default 50) and `maxCallsPerSession` (default 500) bound abuse; over-budget calls return a `429`-shaped error. |
| Pagination | **Does not apply** — the broker returns each API response as-is; the sandbox code handles pagination itself against the upstream API. |
| Input validation | **Applies** — path canonicalisation before allowlist match (reject `..`); header intersected with a safe allowlist (`Authorization` dropped); method allowlist; response content-type and size checks; **API-response content treated as untrusted** (T6 safeguard, ADR-008). Concrete contract (ADR-017): every returned body is delivered as a typed `{ untrusted: true, contentType, body }` envelope carrying a structural untrusted-content marker; raw response text is never auto-fed back into the LLM context (it is returned for the guest's code to handle); where a capability declares a response schema, the broker performs structural extraction/validation before return. The unavoidable residual — injection in content the agent ultimately surfaces — is an explicitly accepted documented risk. |
| Error handling / envelope | **Applies** — typed error shapes returned to Python (`auth_refresh_failed`, `403`, `truncated=true`); wire-level codes/strings are deferred to `/spec`. |
| Configuration | **Applies** — `pysandbox.policy.json` manifest (shippable default + signed workspace override). The WASM guest receives no host environment variables; its capability bundle and run parameters are passed in by the Sandbox Runtime via the host import, and the broker control-IPC endpoint is held only by the trusted Rust host. |
| Health checks | **Applies** — the daemon **is** a long-running per-user service (ADR-023). A local-only liveness/readiness check over the control socket reports daemon health and dependency reachability (IdP JWKS, the backend `obo-broker`); used by the OS service manager (systemd/launchd/Windows service) for restart. Not network-exposed. Exact shape in `/spec`. |
| Migrations | **Does not apply** — no persistent schema; the only persisted artefact is the rotating audit log. |
| Graceful shutdown | **Applies** — capability handles invalidated, in-flight runs terminated, the WASM instance and its linear memory dropped on exit. |
| CORS | **Does not apply** — no browser-facing HTTP surface. If MCP-server mode binds a port, it must be local-only and authenticated (not CORS-exposed). |
| Multi-tenancy | **Does not apply** — single user per daemon (per OS user); OBO tokens keyed by `(user, audience, scopes, providerId)`, no cross-user sharing. |

## Non-functional Requirements

- **Performance targets:** per-run resource limits enforced in-runtime by Wasmtime — CPU via fuel (≈10 s equivalent), wall clock 30 s via an epoch deadline, memory 512 MB via maximum linear memory, stdout/stderr 1 MB each (truncated with a marker). Subprocess spawn and file descriptors do not arise — the WASM guest has no process model and no host-filesystem capability. Capability handles expire ≤5 min.
- **Scale:** single-user, interactive workloads; not designed for sustained high throughput or background jobs (a non-goal). Call volume bounded per run/session.
- **Availability:** best-effort, tied to the daemon (a per-user service); no standalone SLO. The committed backend `obo-broker` service introduces an availability dependency for corporate-API capabilities.
- **Security posture:** untrusted agent + untrusted Python; tokens never enter the sandbox; the **WASM capability model is the load-bearing egress control** (ADR-013) — the guest has no socket, filesystem, or process capability, so egress is structurally impossible and the historical `ctypes`/raw-syscall escape does not apply; opaque short-lived capability handles; host/path/method allowlist; no cross-origin redirect following; defense-in-depth redaction of output; **API-response content treated as untrusted to mitigate prompt injection (ADR-008)**. The boundary is a single cross-platform runtime, so the previous per-OS egress-lockdown question (OQ-7) is closed. Residual risks (a Wasmtime runtime escape (now the sole isolation boundary by choice), RustPython coverage/maturity, the same-UID local-process surface (ADR-024), and the residual prompt-injection surface in the final output) are tracked in [10 — Risks](./10-risks.md).
