# 10 — Assumptions, Open Questions, Risks, Alternatives, Dependencies, Rollout

## Assumptions

| ID | Area | Assumption | Rationale |
|---|---|---|---|
| A-1 | Capability lifetime | Capability handles expire within ≤5 minutes. | Stated default in the source; bounds the value of a leaked handle. |
| A-2 | Consent | Consent decisions are cached in memory per workspace and not persisted. | Stated default; avoids silently broadening grants across sessions. |
| A-3 | Resource limits | Defaults: CPU ≈10 s (Wasmtime fuel), wall 30 s (epoch deadline), memory 512 MB (max linear memory), output 1 MB. No subprocess and no host file descriptors arise — the WASM guest has no process or host-filesystem capability. | Source resource-limits table, mapped onto the Wasmtime capability runtime (ADR-013). |
| A-4 | Response cap | `responseMaxBytes` default 1 MB; over-cap responses truncated with a flag. | Stated default. |
| A-5 | Audit retention | Audit log retained 30 days by default, rotated daily. | Stated default. |
| A-6 | Locale | Narrative output is en-GB. | Workspace default locale (`output-language` layer rule). |
| A-7 | Single user | One user per daemon (per OS user); no concurrent multi-user runs. | Implied by the per-user daemon model. |

## Open Questions

All open questions were decided in revision 0.2.0; **none remain blocking**. Decisions are recorded here and folded into the body. **OQ-7 (per-OS egress lockdown) is now closed by ADR-013** — the isolation boundary is a single cross-platform WASM capability runtime, so there is no per-platform egress mechanism left to verify; the residual is a Wasmtime runtime escape (the top Risk below), not a macOS/Windows gap. **OQ-3 is superseded by ADR-014** — no CPython is bundled; the guest is RustPython compiled to WASM.

## Risks

- **Wasmtime runtime escape (top risk — the sole isolation boundary).** Per ADR-013 the WASM capability runtime is the *only* isolation boundary; by the *WASM-as-sole-boundary* decision no OS sandbox backstops it. A sandbox escape in Wasmtime (or in the host-import shim) would therefore be unmitigated. *Mitigation:* pin and track a current Wasmtime release and its security advisories; keep the host-import surface minimal (one function) and memory-safe (Rust); a third-party pen test targets the runtime and the shim. **M0 spike acceptance checklist (moves SR-7 from ⚠ to ✅):** from inside the guest — (1) `import socket`/`urllib` is unavailable or cannot construct a working socket; (2) no host-import other than the broker call is reachable; (3) attempts to read the host filesystem fail (no WASI fs capability); (4) `os.fork`/process spawn is unavailable; (5) fuel exhaustion, the epoch deadline, and the linear-memory cap each terminate a hostile guest; (6) the pen test attempts a Wasmtime escape and a shim-confusion attack. *No fallback needed:* because the boundary is one cross-platform runtime, there is no per-OS mock-only posture — the gate is "Wasmtime build passes the pen test," uniformly.
- **RustPython coverage / maturity for the in-contract stdlib.** The narrow contract (ADR-014) depends on RustPython faithfully implementing the in-contract surface enumerated in ADR-014 (`json`, `re`, `datetime`, `base64`, `collections`, `itertools`, plus string/dict/list operations and control flow); gaps or behavioural divergence would break legitimate agent code. *Mitigation:* an M0 coverage spike runs a representative REST-orchestration corpus against the RustPython/WASM guest and records unsupported constructs; the supported subset is documented to agent authors as the contract. A RustPython interpreter bug is contained by the WASM boundary (it cannot become host code execution) but remains a correctness risk.
- **Isolation contract: runtime config + artefact + dependency supply chain (AS-9).** The boundary now rests on a correct least-privilege Wasmtime configuration and the integrity of the bundled RustPython WASM artefact and the Wasmtime/RustPython crate tree. A misconfiguration that grants a WASI capability beyond the audited subset (a filesystem preopen, a socket, env/args), a tampered artefact, or a malicious dependency would collapse the boundary. *Mitigation (now firm):* the Wasmtime hardening config is fixed by the runtime and validated fail-closed at startup (**ADR-019** — transient-execution mitigations on, risky proposals off, fuel/epoch/memory bounds, and a **deny-by-default WASI subset** granting only clock/random/captured-stdio — no fs/sockets/env/args, so no ambient authority); the artefact and dependencies are pinned + lockfiled + content-digested, integrity-checked before instantiation, advisory-gated in CI, never dynamically/remotely loaded, and each per-platform service installer is signed (**ADR-018**; per-platform packaging **ADR-022**, so the integrity contract applies independently to every target installer). *Containment (SR-22/SR-23):* even on a runtime/shim compromise or a Wasmtime escape, the broker independently enforces capId/allowlist/budgets and the runtime holds no tokens (ADR-010), so the blast radius is a local foothold — not token theft or capability beyond what was granted. The threat model's §4A (TB2a/TB-RT/TB2b) is the full STRIDE pass on this surface. *Residual verification (out of design scope):* the reproducible/signed build that realises ADR-018 and the Wasmtime-escape pen test.
- **Same-UID local process drives the daemon (the daemon model's central new surface).** The daemon is a per-user process — there is no shared extension-host memory for a co-installed extension to read — but any process running as the same OS user can attempt to connect to its control socket and *drive* it. *Mitigation (ADR-024):* a per-user socket (`0600` / per-user-SID ACL) + a peer-UID check + a per-launch **connection token** gate access; raw tokens never cross the socket; a rogue same-UID client is bounded by consent + allowlist + budgets + audit to misuse of an already-granted capability (ADR-002) — the same authority the user already holds. The one genuinely new security-critical surface; warrants a dedicated security review + pen test. Tracked as SR-24.
- **Prompt injection via API responses (T6).** *Mitigation (now firm — ADR-008 + ADR-017):* responses cross to the guest as a typed `{ untrusted: true, contentType, body }` envelope with a structural marker; raw response text is never auto-fed back into the LLM context; schema-declaring capabilities get structural extraction; allowlist + call budgets apply. **Accepted residual:** injection in content the agent ultimately chooses to surface to the model cannot be eliminated while the system's purpose is to return API data — accepted as a documented residual risk, owner named at Approval. The *control* is fully specified; the *risk* is bounded and accepted, not open.
- **Cross-capability exfiltration via allowlisted writes.** Prompt-injected agent code can read sensitive data through one in-policy capability and `POST` it out through another in-policy capability whose sink is readable beyond the enterprise (a public issue tracker, an externally visible ticket). Both calls are on the host/path/method allowlist, so the egress controls do not stop it — the data leaves through a *sanctioned* door, not a forged one. *Mitigation:* write capabilities to externally-readable sinks are required to set `requireStepUpAuth` (so a human is in the loop before the write), the authoring guide names this as a non-negotiable rule (12 §2), per-run/session call budgets bound bulk exfiltration, and every write is audited. **Accepted residual:** within a single consented, budgeted, audited run the agent can still move data between two capabilities the user authorised — bounded by the user's own granted scope and recorded, not eliminated. Tracked as SR-27 in the threat model.

- **Connection-token disclosure.** Because the connection token sits in a `0600` runtime file readable by the same OS user, a same-UID process can read it and present it (this is the bounded residual above, not a separate escalation). *Mitigation (ADR-024):* peer-UID + the token authenticate "a same-UID process"; the optional first-connect consent for a new client identity narrows it further; eliminating same-UID drive entirely would require per-client OS-keychain-gated credentials (a `/spec` hardening option). The exposure is equivalent to ADR-020's accepted residual on the forked design, not worse.

- **Workstation manifest tampering.** The agent can usually write workspace files, so an unsigned workspace `pysandbox.policy.json` override is attacker-authorable policy — and for direct-provider capabilities (`github`, not routed through `obo-broker`) the workstation broker is the sole enforcement point. *Mitigation (ADR-021):* workspace overrides are honoured only if admin-signed; an unsigned/invalid override is rejected fail-closed to the shipped (signed service-installer) default; the consent dialog shows host + methods + step-up + provider so the user can refuse a surprising grant. Tracked as SR-25.

- **Inbound `id_token` replay (no sender constraint by default).** The `id_token` is the sole authenticator to `obo-broker` (`../obo-broker/` ADR-004) and is a bearer credential: anyone who captures it mid-lifetime — a TLS-intercepting corporate proxy (common in the target enterprises), a misconfigured logger, or off-host malware — can exercise the user's granted capabilities until it expires. Sender-constrained tokens (DPoP/mTLS) are currently *optional*. *Mitigation:* short `id_token` lifetimes, audience restriction, TLS 1.3; **decision deferred to `/spec`** — evaluate making DPoP or mTLS default-on rather than optional, and bounding the maximum accepted token age. Tracked as SR-28 (Partial — design decision open).

- **Loopback redirect interception during sign-in (ADR-029).** The browser auth-code flow returns the authorization code to a `127.0.0.1` listener; another local process could try to race the port or intercept the code. *Mitigation:* PKCE (the code is useless without the daemon-held `code_verifier`), a CSPRNG `state` and `nonce`, binding the listener to `127.0.0.1` only (never `0.0.0.0`), an ephemeral single-use port closed immediately after the redirect, and a short code-exchange window. The `id_token` is captured in the daemon only. The residual is the same-UID class already accepted under ADR-024 (a same-UID process can already read the connection token). *New surface — joins the ADR-024/sign-in security review.*
- **Windows named-pipe peer-auth (ADR-030).** The Windows local-IPC path (named pipe + SID peer check) is new security-critical code with no prior implementation, unlike the battle-tested UDS+peer-UID path. A DACL or SID-comparison error would weaken the client↔daemon boundary on Windows. *Mitigation:* a per-user-SID DACL, `GetNamedPipeClientProcessId`→token-SID equality, and the connection-token gate (identical to Unix); this path is added to the ADR-024 client-auth pen-test scope. Tracked as an extension of SR-24.
- **Degraded first-run trust UX for unsigned installers (ADR-031).** Until signing certs are available, macOS Gatekeeper and Windows SmartScreen will warn (or block) the unsigned/ad-hoc installers, and the install instructions must tell the user to override — itself a habit that trains users to bypass OS protections. *Mitigation:* documented as an explicit interim posture for first-party/internal testing only; signing/notarization is a parameterised build step flipped on the moment certs exist, with no structural change. **Accepted interim residual** (swskipper: no certs yet).
- **Backend OBO service is a high-value target.** Holds privileged downstream tokens for many users. *Mitigation:* confidential-client hardening, least-privilege scopes, its own threat model.
- **Local audit log is user-writable.** "Append-only" is best-effort. *Mitigation:* SIEM/OTLP export to a tamper-evident sink is the authoritative record and is **mandatory in real-credential mode (ADR-016)** — no reachable sink ⇒ mock/non-sensitive-credentials-only.

> The full residual-risk analysis for the end-to-end solution — the per-boundary STRIDE pass, the derived security requirements (SR-1…SR-29), and the accepted residuals (e.g. the SR-12 prompt-injection residual, ADR-017; the SR-24 co-resident-driver and SR-27 cross-capability residuals) — is maintained in the companion [`../threat-model.md`](../threat-model.md). The sign-off readiness checklist and residual-risk register are owned there and in `/spec`/operational documentation as the design matures.

## Alternatives considered

- **Discrete MCP tools (no code execution).** Smaller attack surface (eliminates the sandbox), easier security sign-off, but context-heavy and limited to pre-built operations. Rejected because arbitrary composition/transformation is a requirement (ADR-001).
- **Pass tokens into Python with a normal HTTP client.** Simplest, but exposes tokens through numerous vectors. Rejected as fundamentally insecure (source Appendix A).
- **Remote/server-side Python execution.** Could move the isolation problem off the workstation entirely. Not pursued — the design targets a local per-user daemon; with the WASM capability runtime (ADR-013) client-side isolation is no longer the weak point that motivated this alternative.
- **OS-level isolation as the boundary (Seatbelt / AppContainer / seccomp).** The previous design (ADR-003); rejected in favour of ADR-013 because it was fragile and per-platform — macOS `sandbox-exec` is a deprecated CLI, Windows AppContainer network isolation has documented bypasses, and a `ctypes`/raw-syscall escape from native CPython was an open risk. The WASM runtime removes ambient authority entirely and is identical on all three platforms.
- **WASM plus a thin OS sandbox (defence-in-depth).** Considered: keep the WASM boundary load-bearing and wrap the Rust host in a minimal OS sandbox as a backstop. **Not adopted** — the user chose WASM as the sole boundary for a single mechanism and pen-test target, accepting that a Wasmtime escape has no second layer (the top Risk above).

## Dependencies

- the daemon-owned auth UI, realised as a **browser auth-code + PKCE loopback flow** (ADR-029) — depends on the system browser and the ability to bind a `127.0.0.1` port; and the OS secret store / IdP token cache.
- **OIDC sign-in via generic discovery** and a **public-client (PKCE) registration** in the IdP (a Dex static client for local validation; any standards OIDC IdP in production). No client secret on the workstation.
- An **MCP server library** for the `mcp-stdio` sub-mode (ADR-028) — a Rust MCP/JSON-RPC implementation; an OAuth2 + PKCE client and a browser-launch crate for ADR-029.
- **OS-service + installer tooling** (ADR-030/ADR-031): launchd (macOS) / Windows service registration; named-pipe IPC on Windows; WiX (`.msi`), `pkgbuild`/`productbuild` (`.pkg`), and `codesign`/`notarytool` + `signtool` wired as an optional build step.
- **The backend `obo-broker` service** (OQ-6 resolved; confidential client — ADR-005), now designed at [`../obo-broker/`](../obo-broker/README.md) (revision 0.7.1). Its provider-plugin layer (`../obo-broker/` ADR-009, Provider plugin) is what makes the IdP choice swappable.
- The **Wasmtime** WebAssembly engine (embedded in the Rust Sandbox Runtime) — the single isolation boundary (ADR-013); pinned to a tracked release with security-advisory monitoring.
- **RustPython** compiled to `wasm32` — the guest interpreter (ADR-014); the WASM artefact shipped in the service installer in place of a CPython distribution.
- A Practice Pack supplying the build/test toolchain for `/spec` and `codegen` (OQ-8).

## Rollout and Rollback

- **Phased rollout** (updated for the WASM pivot, ADR-013/ADR-014): M0 spikes (**Wasmtime escape pen test** + **RustPython in-contract-coverage corpus**, replacing the per-OS egress spikes) → M1 controller+broker+IPC (**mock/non-sensitive credentials only** until the runtime escape pen test passes) → M2 Sandbox Runtime (Rust + Wasmtime + RustPython guest, host-import shim) → M3 policy/consent/audit + SIEM export (mandatory in real-credential mode, fail-closed — ADR-016) + response safeguard (ADR-008) + step-up-auth toggle (OQ-1) → M4 MCP + Copilot participant + dry-run preview (ADR-009) → M5 backend `obo-broker` service (**now committed**; designed at `../obo-broker/`) → M6 hardening + pen test → **M7 production agent integration + distribution (ADR-028–031): the `mcp-stdio` MCP front door, the interactive browser auth-code+PKCE sign-in (closing the previously-unbuilt interactive surface), always-on OS-service registration with the resolved Windows named-pipe peer-auth, and the per-platform service installers (unsigned-now, signable-later).**
- **Local validation harness (not shipped):** a Dex container for OIDC, a dummy REST API run as a local service, and IntelliJ + Claude Code driving the `mcp-stdio` front door through the sandbox to the dummy API — the developer-machine acceptance environment for M7.
- **Dark launch / gating:** real credentials are gated behind a passing Wasmtime-escape pen test (one gate, all platforms); capabilities are individually enabled via the policy manifest, and admin policy can pre-deny.
- **Rollback:** stopping the daemon or revoking consent invalidates all in-flight capability handles (next call returns `403`); no persistent schema means no data migration to unwind; the only persisted artefact is the rotating audit log.
