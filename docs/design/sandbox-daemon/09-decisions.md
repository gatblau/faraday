# 09 — Architectural Decisions

Append-only. Decisions are amended via new entries, not by overwriting.

### ADR-001: Code execution over discrete MCP tools
- **Decision.** The agent authors arbitrary Python that calls APIs through `pysandbox_sdk` inside the sandbox, rather than calling one discrete MCP tool per API operation.
- **Context.** The agent must orchestrate calls across pre-approved APIs. Tool-per-operation bloats the context window with one schema per tool and forces an LLM round-trip per step.
- **Consequences.** Compact context, deterministic in-program orchestration (loops/joins/transforms), and a reduced prompt-injection surface (intermediate responses are handled by code, not re-fed to the LLM). The cost: running untrusted code makes the isolation boundary load-bearing and mandatory — a sandbox-escape pen test and explicit acceptance of the remaining residuals. *(The mechanism is since superseded by ADR-013: the boundary is now the WASM capability runtime, not per-OS egress lockdown — but the obligation to have a load-bearing boundary and pen-test it is unchanged.)* Token protection is equivalent either way (the broker holds tokens in both).
- **Alternatives considered.** Discrete MCP tools (more secure, no sandbox surface, but context-heavy and limited to pre-built operations) — rejected because the requirement is arbitrary composition. Hybrid tools-plus-code — rejected, the worst case still runs untrusted code so the full isolation obligation remains.

### ADR-002: Broker-mediated calls; tokens never enter the sandbox
- **Decision.** All authenticated outbound calls are made by the in-host Identity Broker; the sandbox receives only opaque capability handles and sanitized responses.
- **Context.** The agent and the Python it authors are untrusted; a token marshalled into the sandbox can leak via stdout, tracebacks, env dumps, on-disk caches, or a socket to an attacker host.
- **Consequences.** The secret material is never in the untrusted process, eliminating that entire class of exfiltration. Protecting tokens from the **user** is explicitly *not* a goal — the user owns the host process. The defensible boundary is the agent/sandbox.
- **Alternatives considered.** Passing `Authorization: Bearer` into a Python `requests` call — rejected as fundamentally insecure (see source Appendix A).

### ADR-003: OS-level network egress lockdown is the sole egress control
- **Status.** **Superseded by ADR-013** (WebAssembly capability runtime). Retained for history; the egress guarantee no longer rests on per-OS isolation primitives.
- **Decision.** The sandbox's only network path is the broker IPC socket; all other egress is blocked at the OS level (Linux netns/seccomp, macOS sandbox profile, Windows AppContainer/WFP).
- **Context.** The full Python stdlib (including `socket`, `urllib`, `ctypes`) is present, so no in-language module-hiding can be relied upon.
- **Consequences.** The egress guarantee depends entirely on OS-isolation correctness; `pysandbox_sdk` is a convenience, not a boundary. The control is unproven on macOS/Windows (OQ-7) and must be validated by a spike and a pen test before the guarantee is non-provisional.
- **Alternatives considered.** In-process Python restriction (removing modules) — rejected, trivially bypassed via `ctypes`/raw syscalls.

### ADR-004: Opaque capability handles instead of tokens
- **Decision.** The sandbox is given random 128-bit opaque `capId`s, valid only for one run (≤5 min), bound to the issuing extension instance.
- **Context.** Even with brokering, the sandbox must name *what* it may call; that name must carry no credential and be useless if leaked.
- **Consequences.** A leaked `capId` is non-replayable off-device and expires quickly; abuse is further bounded by per-run/per-session call budgets. `scopes` on a capability are advisory — the enforced controls are host/path/method.
- **Alternatives considered.** Scoped short-lived tokens handed to the sandbox — rejected, a token is still a credential that can be exfiltrated and replayed.

### ADR-005: Backend confidential client for token exchange (committed)
- **Decision.** Token exchange for corporate APIs is performed by a server-side backend confidential client (the `obo-broker` service); the extension sends only the user's `id_token`. The identity provider is pluggable, with no default IdP (`../obo-broker/` ADR-017): Okta, Keycloak, or Entra per deployment.
- **Context.** Token-exchange flows (RFC 8693, and equivalents) require a confidential client credential, which a distributed desktop extension (a public client) cannot safely hold. The provider abstraction lives in the companion `../obo-broker/` (ADR-009).
- **Consequences.** Privileged downstream tokens stay off the workstation. Introduces a new committed component, a new availability dependency, and a high-value attack surface that needs its own threat model. **OQ-6 resolved (build the backend)**; the service is now designed at [`../obo-broker/`](../obo-broker/README.md).
- **Alternatives considered.** Token exchange inside the extension — rejected, infeasible without shipping a confidential credential.

### ADR-006: Embed a pinned minimal CPython
- **Status.** **Superseded by ADR-014** (RustPython compiled to WASM). Retained for history; no CPython is bundled.
- **Decision.** Ship a pinned, minimal CPython in the VSIX rather than discovering the user's interpreter.
- **Context.** Reproducibility and integrity of the sandbox runtime depend on a known interpreter and stdlib.
- **Consequences.** Larger package; controlled runtime and read-only stdlib bind. (Recorded in source as the recommendation for Open Question 3.)
- **Alternatives considered.** Rely on `ms-python.python` interpreter discovery — rejected for reproducibility and integrity reasons; retained as the fallback if embedding proves impractical.

### ADR-007: No cross-origin redirect following; `Authorization` not forwarded across hosts
- **Decision.** The broker does not auto-follow 3xx redirects and never re-sends the `Authorization` header to a different host.
- **Context.** A redirect from an allowlisted host to an attacker host would otherwise leak the Bearer token, defeating the design.
- **Consequences.** A 3xx is returned to the caller as-is; cross-origin flows must be re-validated against the allowlist. Closes a direct token-leak vector.
- **Alternatives considered.** Follow redirects with the header attached (HTTP-client default) — rejected as an exfiltration vector.

### ADR-008: Treat API-response content as untrusted (prompt-injection safeguard)
- **Decision.** API-response content returned through the broker is treated as untrusted and passed through a safeguard before reaching the agent (resolves OQ-5 in favour of building a safeguard).
- **Context.** A response from an allowlisted API can carry text crafted to steer the agent (T6); transport sanitisation (control-char stripping, size cap) does not address injected instructions.
- **Consequences.** Adds a broker-side safeguard stage (marking content as untrusted to the agent and/or structural extraction); the exact mechanism is specified in `/spec`. Reduces but does not eliminate the residual injection surface in the final output, which remains a tracked risk.
- **Alternatives considered.** Accept the risk and document only — rejected, the user chose to build a safeguard. Enforce strict response schemas only — retained as a candidate mechanism within this decision.

### ADR-009: Dry-run preview mode
- **Decision.** `run(...)` accepts a `dryRun` flag that returns the API calls a run would make, without executing them (resolves OQ-4).
- **Context.** Users and agents benefit from seeing planned authenticated calls before side effects occur, especially for write capabilities.
- **Consequences.** Adds a preview path that performs static capability resolution only — no token use, no outbound HTTPS, no side effects. Surfaced through the extension API and the Copilot participant.
- **Alternatives considered.** No preview — rejected, it reduces user control and transparency for an agent running arbitrary code.

### ADR-010: Run the Identity Broker as a separate OS process
- **Decision.** Host the Identity Broker (token vault, capability table, outbound calls, audit) in its **own OS process** — the "broker process" — distinct from the VS Code extension host. The Sandbox Controller (in the extension host) reaches it over a control IPC; the sandbox's IPC terminates at the broker process. Tokens live only in the broker process's memory.
- **Context.** All VS Code extensions share one extension-host Node process, so a co-installed malicious extension could read tokens the broker holds in shared memory — defeating trust boundary 1 from inside the host.
- **Consequences.** Co-resident extensions can no longer read broker memory; the central "tokens never exposed" guarantee holds against the in-host threat. Adds a process and a control IPC between the extension host and the broker process (reusing the existing IPC machinery); the sandbox connects to the broker process directly. The IPC endpoint is restricted (UDS mode 0600 / named-pipe ACL) to the intended peers.
- **Alternatives considered.** Keep the broker in the extension host — rejected: leaves the largest hole in the trust model (co-resident extensions read tokens). Keychain-only with no in-memory cache — rejected: tokens must be in memory to make calls, so this does not remove exposure during use.

> **Numbering note.** This (sandbox) HLD allocates no ADR-011 or ADR-012. The per-OS isolation investigations recorded at revisions 0.4.1/0.4.2 (macOS Seatbelt, Windows AppContainer) were folded into the body as amendments to ADR-003 rather than new ADRs, and were then superseded wholesale by ADR-013 (the WASM pivot). ADR-011 and ADR-012 exist only in the companion `../obo-broker/` HLD — a separate register — and are not referenced here. Numbering resumes at ADR-013.

### ADR-013: Sandbox isolation via a WebAssembly capability runtime (supersedes ADR-003)
- **Decision.** Run the agent-authored code inside a **WebAssembly capability sandbox** — a Rust host (the Sandbox Runtime) embedding the **Wasmtime** engine — instead of an OS-isolated native subprocess. The guest is instantiated with **no ambient authority** and exactly **one capability host import** (the broker call shim); a deny-by-default WASI subset (monotonic clock, randomness, and captured stdout/stderr only — no filesystem, sockets, env, args, or process) is permitted so a real interpreter can run, granting no egress (refined by **ADR-019**). The WASM runtime is the **sole** isolation boundary; no OS sandbox (Seatbelt / AppContainer / seccomp) is layered around the host.
- **Context.** Under ADR-003 the egress guarantee rested on per-OS primitives that proved fragile: macOS `sandbox-exec` is a deprecated CLI, Windows AppContainer network isolation has documented bypasses, and a `ctypes`/raw-syscall escape from native CPython was an open pen-test risk (OQ-7, the design's top risk). A capability runtime removes ambient authority entirely: a WASM guest cannot issue a syscall, so there is no socket to open, no file to read, and no process to spawn unless the host links a capability — and the host links only the broker call.
- **Consequences.** Egress becomes *structurally impossible* rather than blocked after the fact; the `ctypes`-escape and subprocess-spawn concerns are eliminated by construction; resource limits move to Wasmtime (fuel, maximum linear memory, epoch deadline); and the boundary is **identical on Linux, macOS, and Windows**, closing OQ-7 and collapsing three platform designs into one. The trust computing base shifts to Wasmtime + RustPython. **New load-bearing residual:** a Wasmtime sandbox escape is now unmitigated by design (no OS backstop) — it is the single pen-test target and the dominant residual risk (see [10 — Risks](./10-risks.md)). The broker still runs out-of-process (ADR-010): the host shim forwards each call over the existing control IPC, so the guest never touches even the IPC socket.
- **Alternatives considered.** Retain ADR-003's OS isolation as the boundary — rejected: fragile and per-platform (deprecated/bypassable mechanisms). **WASM + a thin OS sandbox as defence-in-depth** — considered and **not adopted**: the user chose WASM as the sole boundary for a single mechanism and pen-test target, accepting the no-backstop residual. A Rust microVM (Firecracker-style) or Rust-driven container — rejected: still relies on the OS/hypervisor and does not help on macOS/Windows.

### ADR-014: RustPython guest with a narrow standard-library contract (supersedes ADR-006)
- **Decision.** Execute agent Python as **RustPython compiled to WASM**, not a bundled CPython. The supported guest contract is the **standard-library data-manipulation surface plus the broker SDK** — `json`, `re`, `datetime`, `base64`, `collections`, `itertools`, string/dict/list operations, control flow, and `api.<provider>.*`. Third-party packages, native/C-extensions, and the networking/`os`/`socket` modules are **out of contract**.
- **Context.** The sandbox's purpose is to orchestrate REST calls *through the broker* and transform JSON — it never performs the HTTP itself and has no numerical-computing need. RustPython covers that surface, gives a single-language (Rust) trusted computing base with no CPython C runtime to audit, and its weak spots (networking/`os`/C-extensions) are exactly the surface the threat model wants absent. CPython-on-`wasm32-wasi` was the higher-fidelity alternative but still cannot load native C-extensions and carries the CPython C runtime into the TCB.
- **Consequences.** ADR-006 (bundle a pinned CPython) is retired — no interpreter is shipped as a native binary; the guest is a WASM artefact. The vague "full stdlib present" promise is replaced by an explicit, smaller, more defensible contract (smaller attack surface). **Cost:** agent code that imports a third-party or native package fails by design; this is a deliberate capability narrowing, documented to agent authors. RustPython coverage/maturity for the in-contract stdlib is a tracked risk to be proven by an M0 coverage spike (see [10 — Risks](./10-risks.md)).
- **Alternatives considered.** CPython on `wasm32-wasi` — rejected for this purpose: heavier, drags the CPython C runtime into the TCB, and still no native C-extensions, for fidelity the narrow REST-orchestration contract does not need. Pyodide — rejected: browser-oriented and heavier.

### ADR-015: Extension-side step-up — challenge-driven re-auth and retry (closes SR-3 parent side)
- **Decision.** For a `requireStepUpAuth` capability, when the backend returns `401` with `WWW-Authenticate: Bearer error="insufficient_user_authentication", acr_values="<required>"` (RFC 9470), the Sandbox Controller **pauses the run, performs an IdP step-up sign-in** (VS Code authentication, requesting the challenged `acr`), obtains a fresh `id_token` carrying the elevated `acr`/`amr`, and **retries the original exchange once**. The step-up assurance is conveyed *only* in the `id_token` `acr` claim — never as a caller-supplied request field. This is the extension counterpart to the `obo-broker`'s server-enforced check (`../obo-broker/` ADR-014).
- **Context.** Step-up was decided as a policy-configurable per-capability toggle (OQ-1), and the server side is fully specified (`../obo-broker/` ADR-014: `acr` against `OBO_STEP_UP_ACR_VALUES`, RFC 9470 challenge). What remained open was *how the extension reacts to the challenge* (the parent half of obo-broker OQ-4). The untrusted agent must not be able to assert step-up itself.
- **Consequences.** Sensitive (write) capabilities are gated end-to-end with no caller-forgeable bypass. The agent observes only a one-time pause and a step-up prompt to the **user**; it never sees a token. A failed or declined step-up returns a typed `step_up_required`/`step_up_failed` error to the guest and the run does not proceed. The exact `acr` value is provider-specific and confirmed per deployment.
- **Alternatives considered.** A caller-supplied `stepUpSatisfied` flag — rejected: trivially forgeable by the untrusted agent (mirrors `../obo-broker/` ADR-014's rejection). Silent re-auth without user interaction — rejected: step-up *is* a user-presence/MFA signal, so it must involve the user.

### ADR-016: Audit export to a tamper-evident sink is mandatory in real-credential mode (tightens OQ-2)
- **Decision.** When the deployment is configured to use **real (non-mock) credentials**, a SIEM/OTLP audit export to a tamper-evident sink is **required**, and its reachability is checked **fail-closed at startup**: if no sink is configured or reachable, the deployment runs in **mock / non-sensitive-credentials-only** mode. The local rotating `.jsonl` log is explicitly **non-authoritative** (user-writable); the exported stream is the system of record.
- **Context.** OQ-2 resolved to "provide a *configurable* export." But the local log is user-writable best-effort, so leaving export optional made non-repudiation (SR-9) conditional on each deployment remembering to enable it — a silent way to lose the authoritative record. The design already gates real credentials behind other controls; the audit sink belongs in that same gate.
- **Consequences.** Non-repudiation no longer depends on a deployment-time choice: real-credential operation *implies* an authoritative audit trail. Reuses the existing gating posture (no proven control ⇒ mock-only), so it adds no new mechanism. Cost: an enterprise without a SIEM/OTLP endpoint runs mock-only until it provides one — an intentional constraint.
- **Alternatives considered.** Keep export configurable + document the risk — rejected: it leaves the authoritative-record guarantee to operator discipline. Make the *local* log tamper-evident (e.g. hash-chain) as the record — rejected: it is still user-writable on the user's own host (ADR-002), so it cannot be authoritative against the defended principal.

### ADR-017: Concrete untrusted-content contract; residual injection explicitly accepted (firms ADR-008, closes SR-12)
- **Decision.** Every API/downstream response body returned to the guest MUST carry a **structural untrusted-content marker** (the result is delivered as a typed `{ untrusted: true, contentType, body }` envelope, not as bare text). The harness/agent integration MUST NOT auto-feed a raw response body back into the LLM context; response data is returned for the guest's **code** to handle. Where a capability declares a response schema, the broker performs **structural extraction/validation** before return. The unavoidable residual — that text the agent ultimately chooses to surface to the model can still carry injected instructions — is **explicitly accepted** as a documented residual risk (RR, owner named at Approval).
- **Context.** ADR-008 decided "treat responses as untrusted" but left the mechanism to `/spec`, which kept SR-12 partial. Prompt injection in returned content cannot be *eliminated* while the system's purpose is to return useful API data to an agent; it can be *bounded* by making the untrusted marking concrete and testable and by never silently re-injecting raw text.
- **Consequences.** The control is now fully specified and testable (every returned body carries the marker; a test asserts its presence — see the threat-model SR-12 row). The requirement is met at the control level; the residual is accepted rather than left as an open gap. Code-execution-over-tools (ADR-001) already minimises re-feeding, which this makes explicit.
- **Alternatives considered.** Schema-only responses (reject anything not matching a declared schema) — retained as an *option per capability* within this decision, not mandated globally (it would break free-form API exploration). Stripping/escaping prose in responses — rejected: lossy and unreliable against a determined injection, and the marker + no-auto-refeed stance is the honest control.

### ADR-018: Sandbox runtime/artefact supply-chain contract (closes SR-21 design clause)
- **Decision.** The isolation contract (AS-9) is pinned and verifiable: (1) the RustPython WASM artefact and the Wasmtime/RustPython crate tree are **pinned by version + lockfile and recorded by content digest** in a committed checksum manifest; (2) the bundled WASM artefact's digest is **verified before instantiation** and instantiation **fails closed** on mismatch; (3) **no dynamic or remote module loading** — only the bundled artefact runs; (4) a **dependency-advisory gate** (e.g. `cargo audit`/`cargo deny`) runs in CI and blocks on known-vulnerable or unreviewed crates; (5) the VSIX is **signed**. These are design requirements independent of the build toolchain (OQ-8); only their implementation mechanics defer to `/spec`.
- **Context.** ADR-013/014 made the runtime engine + artefact load-bearing (AS-9). Leaving supply-chain integrity as a "stance" kept SR-21 partial; a tampered artefact or a malicious crate would collapse the boundary regardless of the runtime's correctness.
- **Consequences.** A substituted or tampered guest/engine is detected (digest check) or prevented (no dynamic load); dependency risk is gated continuously; the artefact's provenance is signed. The control is fully specified; the reproducible/signed *build* that realises it is out-of-scope verification (threat-model §6.1a).
- **Alternatives considered.** Trust-on-first-use of whatever WASM is present — rejected: defeats AS-9. Dynamic plugin loading of guests — rejected: reintroduces an untrusted-code-loading path the pinned-artefact rule exists to close.

### ADR-019: Wasmtime hardening configuration (closes SR-23 config clause)
- **Decision.** The Sandbox Runtime instantiates Wasmtime with a fixed hardened configuration: **transient-execution (Spectre-class) mitigations enabled**; **risky WASM proposals disabled** unless the guest needs them (no threads, no multi-memory, no host-FFI); **`consume_fuel` (CPU), an epoch-interruption deadline (wall clock), and a maximum linear-memory bound** all set; **no debug/backtrace detail** that could leak host addresses; and a **deny-by-default WASI subset** that grants the guest **no ambient authority** — only a **monotonic clock**, **randomness**, and a **captured stdout/stderr write** (fd 1/2 into byte-capped buffers); **no filesystem (no preopens), no sockets, no environment, no args, no process spawn**. The single *capability* host import (the broker call shim) remains the guest's **only** egress to the network, the filesystem, or any secret. The configuration is fixed by the runtime, not guest-influenced, and validated fail-closed at startup. Exact Wasmtime/WASI flag names are confirmed against the pinned engine version in `/spec`.
- **Context.** SR-23's resource-bound and no-credentials halves were already met (ADR-013/010); what remained was specifying the concrete side-channel/hardening posture rather than deferring it wholesale to `/spec`.
- **Amendment (2026-06-13, revision 0.3.1).** The original absolute "no WASI capabilities" clause is refined to the **deny-by-default WASI subset** above. Rationale: every production Python-on-WASM interpreter (RustPython and CPython alike) needs a monotonic clock, randomness, and a stdout/stderr sink; a literal zero-WASI guest cannot run real Python (it fails to instantiate against an empty import set). The subset preserves ADR-013's intent — **no ambient authority, egress only via the broker capability** — because clock/random/captured-stdio grant **no** socket, file, environment, or process authority. The isolation invariant is therefore restated as **"exactly one *capability* host import (the broker) + a hardened WASI clock/random/stdio subset"**, not "exactly one import, zero WASI". This unblocks the real RustPython guest (`docs/spec/sandbox-daemon` C12 / build-plan Phase 7).
- **Consequences.** The runtime's hardening is a named design contract, not an implementation afterthought; combined with the runtime holding no credentials (ADR-010), the blast radius of any residual side channel is bounded. Cost: some Wasmtime performance features stay disabled by default — acceptable for the REST-orchestration workload. The WASI surface is now a named, audited subset rather than "none", and is part of the Wasmtime-escape / shim pen-test scope (10 — Risks).
- **Alternatives considered.** Accept Wasmtime defaults — rejected: defaults can enable proposals/diagnostics the threat model would rather exclude, and "default" is not a reviewable contract.

### ADR-020: Broker control-IPC peer authentication, and the bounded co-resident-driver residual (firms ADR-010, closes SR-24)
- **Decision.** The Identity Broker authenticates every control-IPC connection rather than relying on the endpoint ACL alone. Two controls combine: (1) **OS peer-credential verification** — on accept, the broker reads the connecting peer's credentials (`SO_PEERCRED` / `getpeereid` on UDS; `ImpersonateNamedPipeClient`→`TokenUser` SID, compared with `EqualSid`, on the Windows named pipe — never a `GetNamedPipeClientProcessId` PID lookup, which has a reuse race, see `windows-peer-auth.md`) and rejects any peer whose principal does not match the broker's own and any peer that is not the Sandbox Controller or a Sandbox Runtime it launched; (2) a **per-launch capability secret** — at startup the broker generates a 128-bit CSPRNG secret, hands it to the Sandbox Controller it spawns (and the Controller forwards it to each Sandbox Runtime at spawn, over the spawn channel, never via a workspace file or env var the guest can read), and requires it as a connection credential on every IPC open. A `capId` is bound to the issuing broker instance and its per-launch secret, so a handle observed on stdout (and thus visible to the agent) cannot be replayed by a different local process — this is the concrete mechanism behind the "non-replayable outside the issuing extension instance" property asserted in [04 — Domain](./04-domain.md).
- **Context.** ADR-010 moved the broker out-of-process so co-resident extensions cannot read **token memory**. But a UDS `0600` / named-pipe ACL restricts by **UID**, and every co-resident VS Code extension — plus any user-level process — runs as the same UID. Without per-peer authentication, another local process could connect to the broker and *drive* it (request `mintCaps`, or replay a leaked `capId`), obtaining token **use** even though it cannot read token **bytes**. Token use is the same prize as token theft for most attacks, so the boundary ADR-010 claims needs an authentication mechanism, not just an ACL.
- **Consequences.** Arbitrary local processes (other-UID processes, user-level malware not running inside the extension host, processes that never received the per-launch secret) cannot open the control IPC or replay handles. **Bounded residual (explicitly accepted):** a malicious extension co-resident in the *same* VS Code extension host shares memory with the Sandbox Controller and can therefore read the per-launch secret and impersonate the Controller to the broker. Such an extension can invoke the broker **only within the same envelope a legitimate run faces** — per-session user **consent** is still required on first use of each capability, the host/path/method **allowlist** still applies, **call budgets** still apply, every call is **audited**, and **raw tokens never cross the IPC** (only `{capId, verb, path}` out and sanitised JSON back). It therefore cannot exfiltrate credential material; its worst case is consent-gated, budgeted, audited misuse of an already-granted capability — the same authority the user already holds (ADR-002), now named explicitly rather than left implicit in ADR-010. Eliminating it entirely would require moving the Controller out of the shared host too (tracked as a possible `/spec` hardening, not committed here).
- **Alternatives considered.** Rely on the endpoint ACL alone (the ADR-010 wording) — rejected: it authenticates the UID, not the peer, so it does not exclude same-UID co-resident processes. Move the Sandbox Controller into its own process as well — deferred: it shrinks the residual but adds a second IPC and more lifecycle complexity; recorded as a `/spec`-time hardening option, not adopted now. Bind handles to a TPM/secure-enclave key — rejected as disproportionate for a ≤5-minute single-run handle already bounded by consent, allowlist, and budgets.

### ADR-021: Workspace policy overrides must be admin-signed; consent dialog content is specified (closes SR-25)
- **Decision.** The shipped default `pysandbox.policy.json` (bundled in the signed VSIX, ADR-018) is the trusted baseline. A **workspace override** is honoured **only** if it carries a valid signature from an enterprise-admin signing key configured out-of-band (not a workspace file); an unsigned, mis-signed, or schema-invalid override is **rejected fail-closed to the shipped default** — it is never merged and never partially applied. Where no admin key is configured, workspace overrides are ignored entirely and only the shipped default is used. Separately, the **per-session consent dialog** MUST display, for each capability whose first use it gates: the capability id, the resolved **host**, the permitted **methods**, whether the capability **requires step-up**, and the **provider** — so the user authorises a concrete grant, not an opaque name.
- **Context.** The Capability Manifest is the on-workstation authorisation boundary, and the agent generally has write access to workspace files — so an unsigned workspace override is attacker-authorable policy. For **direct-provider** capabilities (e.g. `provider: github`, called by the broker itself — not routed through `obo-broker`), the workstation broker is the **only** enforcement point: there is no server-side derived-copy re-check (A-5 / SR-2 protect only token-exchange providers that traverse `obo-broker`). A tampered manifest could add a capability binding the user's GitHub Bearer to an attacker host, and consent is the last gate — but only if the user can see what they are consenting to. ADR-010/04-domain assumed "signed/admin-controlled workspace policy" without specifying the signature mechanism, the fail-closed merge behaviour, or the consent-dialog content.
- **Consequences.** Manifest tampering on the workstation cannot widen authority: the worst an attacker-written override achieves is rejection back to the shipped default. The consent dialog gives the user the facts needed to refuse a surprising grant (a write method, an unexpected host). **Cost:** enterprises must provision and rotate an admin signing key and a signing workflow for legitimate overrides (the [12 — Authoring Guide](./12-authoring-guide.md) human-approval step feeds this); a deployment with no admin key runs on the shipped default only. The exact signature format and key-distribution channel are confirmed in `/spec`.
- **Alternatives considered.** Merge unsigned workspace overrides over the default (the convenient behaviour) — rejected: it hands manifest control to anyone who can write the workspace, defeating the authorisation boundary for direct-provider capabilities. Trust-on-first-use of a workspace manifest — rejected for the same reason. Rely on consent alone without signing — rejected: consent fatigue plus an attacker-crafted plausible-looking capability is a weak last line; signing removes the untrusted-policy path before consent is even reached.

### ADR-022: Per-platform signed VSIX packaging for the native Sandbox Runtime (resolves the single-distributable tension)
- **Decision.** Distribute the extension as **per-platform signed VSIXes — one per supported `vsce --target`** (`win32-x64`, `win32-arm64`, `darwin-x64`, `darwin-arm64`, `linux-x64`, `linux-arm64`), each bundling the prebuilt Rust **Sandbox Runtime** binary for that target. The RustPython WASM guest artefact and its content-digest (ADR-018) are **identical across every target**; the only per-platform difference is the native host binary.
- **Context.** The Sandbox Runtime is a native Rust binary embedding Wasmtime (ADR-013); a native binary cannot live in one cross-platform package. The prior "single cross-platform distributable" wording (05-tech-stack) is therefore incompatible with shipping a native host. VS Code resolves exactly this with platform-specific extension packages (`vsce package --target <os-arch>`), where the marketplace serves the matching VSIX per client OS/arch.
- **Consequences.** CI builds the Rust host **once per target**, producing a build/signing matrix rather than one artefact. The signed-VSIX + digest-verification integrity contract (ADR-018) applies **independently to each target VSIX** — each is signed, and each carries the same digest-verified WASM guest. The marketplace serves the correct VSIX per client OS/arch; the WASM guest stays single-source. **Cost:** a build/signing matrix instead of a single artefact, and per-target CI.
- **Alternatives considered.** (a) One fat VSIX bundling all platform binaries — rejected: package size and signing overhead, and it ships unusable binaries to every client. (b) Download the native host at runtime — rejected: it violates ADR-018's "no dynamic or remote module loading" and the pinned-bundled-artefact integrity contract.
- **Update (daemon fork, 0.1.0).** Amended by **ADR-023**: the unit of distribution is no longer a VSIX but a **per-OS/arch signed host-service installer** (systemd user unit / launchd agent / Windows service). The per-target build/signing matrix and the ADR-018 digest-verified single-source WASM guest are unchanged; only the package format and install target change (marketplace VSIX → service installer).

### ADR-023: Host-resident per-user daemon; thin IDE-independent clients; single-`run` transport
- **Decision.** Run the Controller, Identity Broker, and Sandbox Runtime as a single **per-user, host-resident daemon** (`faradayd`), not inside a VS Code extension host. Agent hosts are **thin clients** that connect over a **local socket** (UDS `0600` / named pipe with a per-user ACL — never a network transport) through a **single `run(code, requestedCapabilities[, dryRun])` entry**, exposed both as a faraday-native RPC and as **one MCP tool** (never per-API tools). **No per-IDE plugin is built or installed**; an MCP-capable IDE registers the daemon by configuration, a CLI invokes it directly. The daemon multiplexes sessions keyed by `(client identity, workspace)`.
- **Context.** Under the forked design the Controller and the interactive surfaces lived in the VS Code extension host, binding the sandbox to one IDE even though the Broker (ADR-010) and Runtime (ADR-013) were already separate native processes. The goal is a sandbox usable from **any** IDE/agent host with **no per-IDE plugin to build and maintain**, while keeping faraday's thesis (code execution, not per-operation MCP tools — ADR-001) intact.
- **Consequences.** One token vault and one audit point per user, reused by every client; the WASM egress and token-custody guarantees are unaffected (they live in the Runtime/Broker, unchanged). A single `run` tool reintroduces none of ADR-001's objections (one schema; the code orchestrates multi-call work in-process; intermediate responses stay in the sandbox; arbitrary composition) — so MCP-as-transport honours the thesis; only **per-API MCP tools** would betray it, and the daemon exposes none. **New:** a long-lived per-user daemon any same-UID process can attempt to drive (see ADR-024) and a single point of failure for all clients (clients fail closed, never to ambient execution). Amends ADR-010 (the broker is now an in-daemon module, still memory-isolable as a child process) and ADR-022 (packaging → service installer).
- **Alternatives considered.** Keep the Controller in each IDE's extension host and ship per-IDE plugins — rejected: N plugins to build/maintain, the maintenance burden this design exists to remove. A network-exposed daemon — rejected: widens the boundary far beyond the local user; the socket stays local-only. Per-API MCP tools as the surface — rejected: that is exactly the pattern ADR-001 replaces.

### ADR-024: Local client authentication — connection token + peer-UID + first-connect consent (amends ADR-020)
- **Decision.** A client authenticates to the daemon by three combined controls: (1) the socket is **per-user** (`0600` UDS / per-user-SID named-pipe ACL); (2) on accept the daemon verifies the **peer principal** equals its own (`SO_PEERCRED`/`getpeereid` on Unix; on Windows the impersonated client `TokenUser` SID via `ImpersonateNamedPipeClient`, compared with `EqualSid` — never `GetNamedPipeClientProcessId`, which has a PID-reuse race) and rejects any other; (3) the client must present a **connection token** — a 128-bit CSPRNG secret the daemon writes to a `0600` runtime file at startup and the client reads. Optionally (defence-in-depth), the **first connection of a new client identity** raises an `interaction_required` consent before any `mintCaps`. This replaces ADR-020's per-launch secret, which was distributed through a spawn relationship the daemon no longer has.
- **Context.** ADR-020 stopped *other same-UID processes* from driving the broker via a secret handed down the spawn tree. A long-lived daemon does not spawn its clients, so that mechanism cannot carry over; without a replacement, "any same-UID process can drive the daemon" — a wider surface than the co-resident-extension residual ADR-020 accepted.
- **Consequences.** Controls (1)+(2)+(3) authenticate "a same-UID process that can read the connection token" — the **same class of authority** ADR-020 already accepted (a same-UID process could read the per-launch secret), so the residual is comparable, not worse; (4) optionally narrows it. Raw tokens still never cross the socket (only `{capId, verb, path}` out, sanitised JSON back); consent, allowlist, budgets, and audit still bound a rogue same-UID client to consent-gated, budgeted, audited misuse of an already-granted capability (ADR-002). **This is the one genuinely new security-critical surface; it warrants a dedicated security review and pen test before production.**
- **Alternatives considered.** Peer-UID only (no token) — rejected: loses the ADR-020-equivalent narrowing. A network auth scheme (OAuth/mTLS between client and daemon) — rejected: disproportionate for a same-host, same-user local socket. Per-client OS-keychain-gated capability — recorded as a `/spec`-time hardening option, not required for the baseline.

### ADR-025: Interaction delegation via a daemon-owned consent/auth UI (amends ADR-015, ADR-021)
- **Decision.** When sign-in, per-session consent, or step-up is required, the daemon returns a typed **`interaction_required`** message (`sign_in` / `consent` / `step_up`, carrying the ADR-021 consent content and the RFC 9470 step-up parameters). The interaction is rendered, in order of preference, by (1) a **daemon-owned minimal UI** — tray applet / native dialog / short-lived local `127.0.0.1` page (the default; transport-agnostic; needs no IDE plugin); (2) **MCP elicitation**, where the connecting client supports it; or (3) the **CLI** in-terminal. The daemon remains the source of truth for *what* is required and *whether* it was satisfied; the renderer supplies only the UI. The agent never asserts any of these.
- **Context.** ADR-015/ADR-021 rendered sign-in/consent/step-up in the VS Code extension host. A headless daemon has no UI, and the maintenance goal forbids depending on a per-IDE plugin to provide one.
- **Consequences.** Interactive auth works with the MCP or CLI front door and **no IDE plugin**. A fully headless caller (no UI, no elicitation) runs in a **non-interactive posture** — pre-granted consent via signed policy (ADR-021) and/or mock / non-sensitive-credentials mode (ADR-016); a sensitive (write) capability with real credentials always requires a human-present renderer, else the call fails closed. The untrusted-signal stance of ADR-015/ADR-021 is preserved end-to-end.
- **Alternatives considered.** Require each client to render prompts — rejected: re-introduces the per-IDE-plugin burden. Daemon performs silent re-auth — rejected: step-up *is* a human-presence signal (mirrors ADR-015's rejection).

### ADR-026: Implement the entire daemon in Rust (single binary)
- **Decision.** Implement the whole daemon — Controller, Identity Broker, Sandbox Runtime, client-authentication, the consent/auth UI, and the control-socket/MCP front door — in **Rust**, as one binary. This supersedes the forked design's TypeScript Controller/Broker; the Runtime was already Rust (ADR-013/ADR-014).
- **Context.** The daemon is a single long-lived, security-critical, credential-holding host service. A single-language Rust implementation gives one memory-safe trusted computing base, a lean and auditable dependency tree (`cargo audit`/`cargo deny`, ADR-018), a small static binary suited to a per-user service, and removes the Node/TypeScript runtime and its supply-chain surface entirely. The Wasmtime engine and RustPython guest are already Rust, so the Runtime needs no FFI bridge to a separate host language.
- **Consequences.** No Node/npm in the trusted path; one toolchain (`cargo`) for build/test/lint/audit; the consent UI uses a Rust-native approach (a minimal local HTTP page served by the daemon, or a Rust GUI/tray crate). **Cost:** the team needs Rust competence across the whole daemon, not just the Runtime; some IDE-integration glue that was trivial in a TS extension becomes a thin per-IDE config or a small plugin (optional, ADR-023). Mature Rust crates exist for the load-bearing parts (Wasmtime, OIDC/OAuth2 clients, HTTP, TLS, KMS/keychain), so no security-critical logic is hand-rolled.
- **Alternatives considered.** Keep TypeScript for the Controller/Broker and Rust only for the Runtime — rejected: drags a second runtime and the npm supply chain into a credential-holding daemon, for no benefit once the extension host is gone. Go for the Controller/Broker (as the obo-broker uses) — viable and memory-safe, but it splits the daemon across two languages and an FFI boundary to the Rust Runtime; Rust keeps it one binary, one TCB.

### ADR-027: Observability scope for the per-user dev-machine daemon (lean; OTel optional)
- **Decision.** For the per-user, host-resident **dev-machine profile**, the daemon's observability is the **append-only audit log** (C3 — per outbound call: keyed-HMAC user id, provider, host, path, method, status, sizes, `durationMs`; never tokens/bodies; OTLP-exportable and **mandatory in real-credential mode**, ADR-016) plus **structured, redacted logging** (XC3, `run_id`-correlated). A full **OpenTelemetry metrics + traces** pipeline (the LLD's earlier XC4/XC5) is **out of scope by default** and, if ever needed, an **optional, off-by-default** capability for fleet deployments.
- **Context.** OTel metrics/traces earn their value when a *fleet* is aggregated into a central collector (dashboards, cross-service tracing). On a single developer's machine there is no fleet, no default collector, and `run_id` already correlates the audit + logs locally; the one remote hop (`obo-broker`) carries its own server-side telemetry. The full OTel SDK + OTLP exporter would also expand the dependency tree against the lean single-binary, auditable-supply-chain goal (**ADR-026**). The HLD already framed metrics as "no separate metrics pipeline" and tracing as "`runId` correlation" — this ADR records the decision explicitly and re-aligns the LLD (XC4/XC5) to it.
- **Consequences.** Observability that matters locally — who did what (audit, compliance/non-repudiation) and what happened in a run (redacted logs, `run_id`) — is fully present; no OTel/OTLP/gRPC dependency is taken. **Cost:** no out-of-the-box fleet metrics/traces; a fleet operator who wants them enables the optional OTel layer (FU-028), which must preserve the cardinality bounds (no `path`/`user`/`run_id` labels) and the never-log-a-token guarantee.
- **Alternatives considered.** Build full OTel now (the earlier XC4/XC5) — rejected for the dev-machine profile: supply-chain weight for marginal local value. Minimal in-process counters surfaced via the local status endpoint — viable middle ground, deferred to FU-028 if demand appears.

### ADR-028: MCP front door realised as an stdio sub-mode of the daemon binary (concretises ADR-023/ADR-025)
- **Decision.** The "single MCP tool" front door of ADR-023 is realised as a sub-mode of the **same `faradayd` binary** — `faradayd mcp-stdio` — that an MCP client (Claude Code / any MCP-capable IDE) launches **per session** and speaks the **MCP protocol** (JSON-RPC 2.0: `initialize` / `tools/list` / `tools/call`) to over **stdin/stdout**. It exposes exactly one tool, **`python_sandbox`** (`{ code, requestedCapabilities[, dryRun] }`). The sub-mode is an **untrusted client** of the persistent daemon: it reads the user's `0600` connection-token file, connects to the daemon's local control socket (ADR-024), and relays `{code, requestedCapabilities}` → sanitised result. It **holds no tokens**; the `id_token` and provider tokens never reach it (they stay in the daemon — ADR-002/ADR-010). One binary (ADR-026), so the sub-mode and daemon are always version-matched.
- **Context.** ADR-023 committed to "one MCP tool" but left the transport open. MCP has two transports: **stdio** servers are per-client subprocesses the client spawns and owns; **HTTP/SSE** servers are listeners. The daemon is a single, always-on, per-user service that owns the socket, token file, sessions, capability table, audit log, and the warm Wasmtime engine + guest (ADR-023/ADR-026) — so it **cannot itself be** a client-spawned stdio subprocess. stdio therefore needs a thin per-session front process; HTTP would let the daemon listen directly.
- **Consequences.** MCP clients integrate by **pure configuration** (`command: faradayd, args: [mcp-stdio]`) — no per-IDE plugin (ADR-023). The token-custody daemon keeps its minimal **local-only** surface (no network port; ADR-024 unchanged). Untrusted **MCP parsing runs in the throwaway per-session process, outside the credential-holding TCB**. **Cost:** one extra local process hop (negligible latency) and a small MCP-server code path in the binary. The sub-mode is itself a "client" under ADR-024 and authenticates exactly as any client does.
- **Alternatives considered.** **MCP-over-HTTP/SSE in the daemon** — rejected for the per-machine profile: opens a loopback TCP port (not peer-UID-gated the way a `0600` socket is, so it would need a re-invented bearer-auth) and pulls untrusted MCP parsing into the token-custody process; recorded as the option to revisit only if remote / multi-machine clients become a requirement. **A separate standalone bridge binary** — rejected: two artefacts to build/sign/version; a sub-mode of the one binary keeps the sub-mode↔daemon versions locked and the supply chain lean (ADR-026).

### ADR-029: Interactive sign-in via browser authorization-code + PKCE on a loopback redirect (concretises ADR-025; closes the daemon-owned-UI gap)
- **Decision.** The daemon-owned interactive surface (ADR-025) realises OIDC sign-in as the **OAuth 2.0 authorization-code flow with PKCE** (RFC 8252 native-app pattern): the daemon starts a **transient `127.0.0.1:<ephemeral>` HTTP listener**, opens the user's **system browser** to the IdP `authorize` endpoint (with `code_challenge`, `state`, `nonce`, and the loopback `redirect_uri`), receives the redirect on the loopback listener, exchanges the code for tokens at the IdP token endpoint, and **captures the `id_token` in the daemon only**. The same surface renders per-session **consent** (ADR-021 content) and **step-up** (ADR-015 / RFC 9470 `acr_values`). It targets **generic OIDC discovery** (`/.well-known/openid-configuration`), so any standards-compliant OIDC IdP works; **Dex** is one such instance used for local validation. This replaces the unbuilt headless placeholder.
- **Context.** ADR-025 chose a daemon-owned UI ("tray / native dialog / 127.0.0.1 page") but committed no concrete sign-in mechanism, and the implementation shipped a **headless placeholder that fails sign-in closed** — so real interactive auth was unbuilt, the central remaining gap before production. For a local-desktop per-user daemon, browser auth-code + PKCE on loopback is the standard native-app OAuth pattern, needs no embedded webview, and keeps the `id_token` off every client.
- **Consequences.** Real user sign-in works with **no IDE plugin**; the `id_token` is captured and held only in the daemon (ADR-002/ADR-010). **PKCE + `state` + `nonce` + `127.0.0.1`-only redirect + an ephemeral single-use port** bound the redirect-interception surface. Step-up reuses the same flow with the challenged `acr`. **Cost:** the daemon must be able to open a browser and bind a loopback port — true for the **committed local-desktop topology**. Remote/SSH/devcontainer sessions (no local browser/loopback) are **out of scope** for this flow; **device-code is the recorded fallback** if that topology is later required.
- **Alternatives considered.** **Device-code flow** — rejected for the committed local-desktop topology (clunkier UX; only needed when there is no local browser/loopback). **Embedded webview collecting credentials in the daemon's own UI** — rejected: phishing-shaped, larger surface, and pulls user credentials through the daemon instead of the IdP's trusted browser session. **MCP elicitation / CLI prompt** (ADR-025 alternates) — retained as optional secondary renderers, not the primary sign-in mechanism.

### ADR-030: Always-on OS-service integration and cross-platform local IPC; resolves the Windows peer-auth question (concretises ADR-023/ADR-024)
- **Decision.** The daemon runs as a **per-user, always-on OS service**: a **launchd user agent** on macOS (`LaunchAgents` plist, `RunAtLoad` + `KeepAlive`) and a **per-user Windows service / login-launched process** on Windows. The control endpoint's local IPC is per-OS: a **`0600` Unix domain socket with `SO_PEERCRED`/`getpeereid` peer-UID** on macOS/Linux (unchanged), and a **named pipe (`\\.\pipe\…`) with a per-user-SID DACL and an `ImpersonateNamedPipeClient`→`TokenUser`-SID (`EqualSid`) peer check** on Windows. The connection token (ADR-024) is presented over either transport identically. This **resolves the previously-open "Windows peer-auth" question**: the Windows peer principal is the impersonated client's token SID, required to equal the daemon's user SID — the exact analogue of UID-equality on Unix. (The check impersonates the connection, not a PID lookup — `GetNamedPipeClientProcessId` has a reuse race; see `docs/spec/sandbox-daemon/windows-peer-auth.md`.)
- **Context.** ADR-023/ADR-024 specified the UDS + peer-UID model and named "named pipe with a per-user ACL" for Windows but left the concrete Windows peer-authentication mechanism open (a non-blocking open question). Production on Windows needs it resolved, and the daemon must be **always-running** for the `mcp-stdio` sub-mode (ADR-028) to find it on each client spawn.
- **Consequences.** The daemon survives logout/restart under the OS service manager (auto-start at login, restart on crash); the always-on guarantee the `mcp-stdio` sub-mode relies on holds. The local-only, peer-authenticated, token-gated boundary (ADR-024) is now **concrete and equivalent on both platforms** (SID-equality ≈ UID-equality). **Cost:** per-platform service-registration and IPC code; the **named-pipe path is new security-critical surface** and joins the ADR-024 client-auth review / pen-test scope. The `mcp-stdio` sub-mode must emit a clear, actionable error when the service is not running.
- **Alternatives considered.** **A network/loopback control socket** to avoid per-OS IPC differences — rejected (ADR-024: local-only, not network-exposed). **A one-shot launch per request** instead of an always-on service — rejected: defeats the warm-engine / persistent-session / single-audit-point rationale of ADR-023 and would cold-start the ~16 MB guest + engine on every call.

### ADR-031: Distribution as per-user service installers; unsigned-now with signing/notarization a parameterised build step (supersedes ADR-022)
- **Decision.** Distribute the daemon as a **per-platform service installer** that (1) drops the single binary, (2) registers the OS service (ADR-030), and (3) **writes/merges the MCP client configuration entry** (`command: faradayd, args: [mcp-stdio]`) for detected clients: a **macOS `.pkg` (and/or `.dmg`)** and a **Windows `.msi` (WiX)**. **Code-signing and notarization are an optional, parameterised build step** (cert material injected via CI/env secrets when available); by **default the build emits unsigned / ad-hoc artefacts and requires no certificates**. This **supersedes ADR-022**'s per-platform signed VSIX (a VS Code artefact, obsolete once the daemon is IDE-independent — ADR-023).
- **Context.** ADR-022 predates the daemon model and packaged a VSIX; ADR-023 already noted "packaging → service installer". The artefact-integrity contract (ADR-018, the content-digest pin of the embedded guest) still applies to whatever ships. The user holds **no signing certificates yet**, but production will eventually require them — so signing must be a **switch, not a rewrite**.
- **Consequences.** A developer can install and run the daemon + MCP integration from an **unsigned package today** (Gatekeeper / SmartScreen will warn — acceptable for first-party/internal testing). When certs arrive, flipping the signing step on yields Developer-ID-signed + notarized macOS artefacts and Authenticode-signed Windows artefacts with **no structural change**. The embedded-guest digest pin (ADR-018) is unchanged. **Cost:** an unsigned default has a degraded first-run trust UX (documented — see Risks); the installer must **merge, not clobber**, existing MCP client config.
- **Alternatives considered.** **Require signing from day one** — rejected: blocks all progress on certificates the user does not hold. **Ship a bare binary + manual instructions** — rejected: poor UX and easy to misconfigure the service + client config; an installer encodes the service registration and config-merge correctly once.

### ADR-032: Dev-only plaintext egress to loopback providers, gated and fail-closed (relaxes the C10 HTTPS-only invariant; mirrors ADR-029)
- **Decision.** The DownstreamClient (C10) selects the egress scheme **per call** from the resolved capability host: `https` by default, and `http` **only when both** (1) the operator sets `PYS_ALLOW_PLAINTEXT_LOOPBACK_EGRESS=true` (default `false`) **and** (2) the host is the literal loopback IP `127.0.0.1` (matched exactly, or as `127.0.0.1:<port>`). `localhost` does **not** qualify — the host must be the loopback IP, so a DNS name (including a trick like `127.0.0.1.evil.com`) can never satisfy the rule. A remote `http` target is **never** producible — no configuration downgrades egress to a non-loopback host. This mirrors the loopback-only plaintext exception already granted to the OIDC issuer (**ADR-029**) and relaxes, in a bounded way, the prior C10 invariant that *"production never has an `http` code path"*.
- **Context.** C10 was HTTPS-only in production; its plaintext path existed solely under the `integration` cargo feature (test stubs). The developer-machine demo (docker-compose: go-httpbin on `127.0.0.1:8080`, plaintext) therefore could not complete a live `dummy` call against the installed binary — the daemon dialled `https://127.0.0.1:8080` against a plaintext listener and failed `DOWNSTREAM_UNAVAILABLE`. ADR-029 already established that loopback plaintext is acceptable for the local dev topology (the Dex IdP) with remote `http` rejected; egress lacked the equivalent rule.
- **Consequences.** The demo works end-to-end on loopback with an explicit opt-in; the production default (`false`) is unchanged — HTTPS-only and fail-closed. **Risk made explicit:** a `passthrough` capability forwards the user's OIDC `access_token` as a `Bearer` header (C11→C10); over plaintext that token is in cleartext on the wire. The loopback-only constraint bounds this to the local host (no network egress), and default-off + opt-in + remote-`http`-rejected prevents the token ever reaching a remote host in cleartext. The toggle **MUST NOT** be enabled on a host whose loopback interface is shared (multi-tenant / shared-kernel). **Cost:** C10 gains a small, bounded `http` branch in production code (no longer feature-gated out), which joins the ADR-024 / AS-17 egress review scope.
- **Alternatives considered.** **A general "disable TLS" switch** — rejected: would permit cleartext token egress to remote providers; the loopback constraint *is* the security boundary, not an optional nicety. **Front the demo with a TLS reverse proxy on a trusted cert** — most faithful to production but heavier dev setup and a cert-trust problem (`reqwest` validates; a self-signed cert needs trust wiring); retained as the recommended path for a working live call *without* enabling plaintext. **Keep C10 HTTPS-only and document the demo limitation** — remains the posture whenever the toggle is off; rejected as the only option because the user wants a working live `dummy` call.

### ADR-033: Resource-audiencing for pass-through — the daemon requests a per-capability resource audience at sign-in (concretises ADR-029; makes pass-through's "audienced for the provider" premise enforceable)
- **Decision.** For a `Passthrough` capability, the daemon **requests the capability's `audience` at sign-in** so the IdP issues an access token whose `aud` names the **resource server**, which then validates it before serving. The Controller (C13) collects the distinct `audience` values of a run's resolved capabilities and passes them in `InteractionRequired::SignIn`/`StepUp`; the ConsentUI (C8) adds a resource-audience request to the authorize/token call. The mechanism is the IdP's: for the demo Dex it is the **cross-client trusted-peer** scope `audience:server:client_id:<resource-client-id>`; for a generic OIDC IdP the standards-track equivalent is the **RFC 8707 `resource`** indicator. The `audience` field already exists on `ResolvedCapability`; this decision only makes the daemon *consume* it. The held `access_token` (already forwarded by C10/C11) is unchanged in handling — it is now merely audienced correctly.
- **Context.** Pass-through (C10) is only safe when the resource server validates a token *audienced for it* (the broker says so at `broker.rs:290-292`). Until now the daemon forwarded the user's access token but never asked the IdP to audience it for the resource, and the demo's stub service validated nothing — so the premise was never exercised (RFC 01). Without a requested audience, a Dex access token is audienced for the `faradayd` client, not for any resource, so a correctly-implemented resource server would reject it.
- **Consequences.** The premise becomes enforceable end-to-end: a validating resource server (RFC 01) accepts only a token carrying its audience, and the daemon makes that token obtainable. **Cost / surface:** the sign-in challenge type gains an `audiences` field (Phase 2C), and the IdP must be configured to grant the audience (a Dex `trustedPeers` entry on the resource client for the demo). A capability whose `audience` the IdP will not grant fails closed at sign-in (`SIGN_IN_FAILED`). The OBO `exchange` path is unaffected — it audiences downstream tokens server-side via the obo-broker. **Out of scope:** changing how the access token is *applied* (still a `Bearer`, C10) or the `exchange` path.
- **Alternatives considered.** **Resource server accepts `aud == faradayd` (the client)** — simpler, demo-only, but validates "a token issued to the faraday client" rather than "a token for this resource"; rejected because it does not exercise the actual premise (RFC 01, Option A chosen). **A single daemon-level audience env var** — rejected: audiences are per-resource, so they belong on the capability, not on one global setting. **RFC 8707 `resource` only** — retained as the generic-IdP path but not the demo default, because Dex's first-class mechanism is the trusted-peer audience scope (OQ-G).
