# 09 — Architectural Decisions

Append-only. Decisions are amended via new entries, not by overwriting.

### ADR-001: Exist as a server-side confidential client
- **Decision.** Perform the token exchange in a server-side service holding the confidential-client credential; the daemon sends only the user's `id_token`.
- **Context.** Token-exchange flows (Okta RFC 8693, Microsoft OBO, and equivalents) require a confidential client; a distributed desktop daemon (public client) cannot safely hold one (parent `../sandbox-daemon/` ADR-005). The concrete identity provider is now pluggable (ADR-009); this decision holds for whichever provider is configured.
- **Consequences.** Privileged downstream tokens never reach the workstation. Introduces a new deployable, an availability dependency, and a high-value attack surface — addressed by this HLD and a dedicated threat model/pen test.
- **Alternatives considered.** OBO inside the daemon — rejected, infeasible without shipping a confidential credential. Hand the user a downstream token — rejected, replayable and defeats the sandbox boundary.

### ADR-002: Single-tenant per enterprise deployment
- **Decision.** Each enterprise runs its own instance with its own confidential-client credential, cache, and audit.
- **Context.** The service holds many users' privileged access; cross-organisation isolation is a first-class security concern (interview decision).
- **Consequences.** Strong isolation by construction; no in-process tenant partitioning needed. Higher per-tenant operational footprint; multi-region/DR handled per deployment.
- **Alternatives considered.** Multi-tenant SaaS — rejected for the initial design; cross-tenant isolation risk outweighs the operational saving for enterprise customers.

### ADR-003: Distributed, encrypted token cache
- **Decision.** Cache exchanged downstream tokens in a distributed cache (assumption: Redis), encrypted at rest (AES-256-GCM under KMS-derived per-entry data keys — ADR-011), keyed by `(user, audience, scopes, providerId)` with TTL ≤ token expiry.
- **Context.** The service is horizontally scaled and stateless; tokens must survive instance restarts and be shared across replicas (interview decision).
- **Consequences.** Horizontal scale without sticky routing; an added infrastructure dependency and an encryption-key custody requirement.
- **Alternatives considered.** In-memory per-instance cache — rejected; loses tokens on restart and needs sticky sessions, harming availability.

### ADR-004: Authenticate requests via audience-restricted user id_token
- **Decision.** Trust an incoming request only after validating the user's `id_token` (issuer, signature, expiry, `audience` = this service); require no client secret from the daemon. The provider-specific validation (which issuer/JWKS) is delegated to the active Provider Plugin (ADR-009).
- **Context.** The daemon is a public client with no secret to present (interview decision); the issuer is whichever IdP a deployment configures (e.g. Okta, Keycloak, Entra — no default, ADR-017).
- **Consequences.** Simple, credential-free client integration; security rests on correct token validation and audience restriction. Optional transport hardening (mTLS) remains available as a non-blocking enhancement.
- **Alternatives considered.** Require a client credential / mTLS from the daemon — deferred as an optional enhancement; adds deployment friction without a public-client secret.

### ADR-005: Deploy on self-managed Kubernetes
- **Decision.** Package as a container and run as a stateless Kubernetes Deployment + Service.
- **Context.** Interview decision; existing self-managed cluster operations.
- **Consequences.** Standard scaling, health probes, and rollout/rollback; the team owns cluster operations and credential delivery (workload identity).
- **Alternatives considered.** Managed PaaS (App Service / Container Apps) — viable, lower ops, not chosen. Serverless functions — rejected, cold-start and token-cache sharing complicate the OBO flow.

### ADR-006: Confidential-client credential via certificate and workload identity
- **Decision.** Use a certificate (preferred over a shared secret) loaded through Kubernetes workload identity / an external secret store, with rotation.
- **Context.** A static secret on disk is a custody liability for a high-value service.
- **Consequences.** No static secret at rest; supports rotation without redeploy. Requires workload-identity federation to be configured in the cluster.
- **Alternatives considered.** Static client secret in a Kubernetes Secret — retained as a fallback if workload identity is unavailable, with reduced custody guarantees.

### ADR-007: Never return downstream tokens; no cross-origin redirect following
- **Decision.** Return only the sanitized downstream response; never the downstream token. The downstream HTTP client does not auto-follow cross-origin redirects and never forwards `Authorization` across hosts.
- **Context.** Carries forward the parent design's token-leak prevention (`../sandbox-daemon/` ADR-007) to the server boundary.
- **Consequences.** Closes the redirect-based token-leak vector and keeps privileged material server-side.
- **Alternatives considered.** Return the token for the daemon to use — rejected, recreates the exfiltration risk the whole design removes.

### ADR-008: Build a thin service on off-the-shelf plumbing, rather than adopt an API gateway or IdP token-exchange product
- **Decision.** Build `obo-broker` as a thin bespoke service that owns only the contract — capability-policy enforcement, token-exchange orchestration via the active provider plugin, the token-cache adapter, and the never-return-token guarantee — and offload every generic cross-cutting concern (TLS, rate limiting, optional mTLS, credential custody, encrypted cache, audit transport) to existing cluster infrastructure. Do **not** adopt a full IdP (Keycloak/Curity) or an API gateway (Azure API Management) as the primary mechanism.
- **Context.** The constraints set in this HLD decide it: self-managed Kubernetes (ADR-005), single-tenant per enterprise (ADR-002), and a specific contract (single-call proxy, never return the token, capability handles). The novel security code is minimal — the hard parts are battle-tested libraries (JWT validation, standard OIDC/OAuth2 clients) and platform primitives, not bespoke crypto — so "build" here is mostly glue and policy enforcement, which is the easiest part to audit and pen-test.
- **Consequences.** Smallest auditable surface enforcing the exact contract; light to operate per tenant; no new Azure control-plane dependency. Generic concerns ride on the cluster ingress (e.g. Envoy) + workload identity + the encrypted cache + OTel/SIEM rather than new code. Cost: it remains bespoke code that requires its own security review and pen test (already mandated in [07 — Cross-cutting](./07-cross-cutting.md) and [10 — Risks](./10-risks.md)).
- **Alternatives considered.** *Azure API Management* — rejected as the primary mechanism: re-adds an Azure control-plane dependency that conflicts with the self-managed-Kubernetes choice, and its permissive-by-default policy engine would still need custom policy to enforce the never-return-token contract. *Keycloak / Curity token exchange* — rejected: a full IdP is heavy to operate per tenant. **Revisit triggers:** a move to managed Azure (reconsider APIM credential manager); the organisation already operating Keycloak/Curity (reuse it instead of writing the exchange).
- **Update (rev 0.3.0).** ADR-008's argument against non-MSAL languages was "don't forfeit first-party MSAL." With a provider-pluggable, standards-based model (ADR-009) and Okta as the first provider, that constraint dissolves — each plugin uses whatever mature, well-supported library fits its IdP (Okta SDK / standard OIDC + OAuth2 libraries). The "use a vetted library, don't hand-roll the security-critical exchange" principle still governs the language choice.
- **Update (rev 0.8.0).** Two clarifications for ADR-017's provider-neutral baseline. (1) **Keycloak role disambiguation.** This ADR rejects **Keycloak/Curity as the broker** — operating a full IdP as the primary exchange mechanism. It does **not** reject **Keycloak/Curity as identity providers the broker performs RFC 8693 token exchange against**: those are supported peer plugins (ADR-009/ADR-017), since Keycloak implements RFC 8693 token exchange. The two roles are distinct and both stances stand. (2) **Okta-portfolio evaluation.** A later evaluation tested whether any Okta-portfolio product — API Access Management, Cross-App Access, Auth0 Token Vault, or Okta Access Gateway — could replace the broker. It reached the same conclusion as this ADR's original gateway/IdP analysis: each either returns a downstream token to the caller or does not proxy-and-sanitize, so none satisfies the never-return-token contract (ADR-007). The broker is built and kept; see [10 — Risks](./10-risks.md) Alternatives. **Revisit trigger:** a deliberate decision to relax the never-return-token guarantee (e.g. accept a scoped on-workstation token via Token Vault) would amend ADR-001/ADR-007.

### ADR-009: Provider plugin architecture for identity and credential acquisition
- **Decision.** Make the broker core provider-agnostic. All identity-provider-specific concerns — validating the inbound user token and acquiring + attaching the downstream credential — are encapsulated behind a **Provider Plugin** interface, selected per capability by a **Provider Registry**. Ship an Okta token-exchange plugin first; Entra OBO, GitHub OAuth, an Okta access-gateway (header-injection) plugin, and a generic RFC 8693 plugin are additional in-tree implementations.
- **Context.** The organisation uses Okta, but the exact fronting model (token exchange vs access gateway) is not yet pinned, and future deployments may use a different identity provider. Hard-coding Microsoft (the original draft) already proved brittle; a plugin boundary keeps the core stable and defers IdP/fronting specifics to configuration.
- **Consequences.** The core (request handling, policy, cache, proxy, never-return-credential, audit) is stable and IdP-agnostic; each provider is an isolated, separately testable, separately security-reviewed unit; the "which Okta flavour" question becomes a plugin-selection detail, not a core redesign; the downstream credential generalises from "access token" to a credential that a plugin applies as a Bearer, custom headers, mTLS, or API key. Cost: a versioned plugin SPI must be designed and maintained. **Trust boundary:** plugins are **first-party, in-tree, compiled-in, and security-reviewed** — never dynamically-loaded third-party code; loading untrusted plugins into a credential-handling service is explicitly out of scope.
- **Alternatives considered.** Hard-code a single IdP (Okta, or the original Microsoft) — rejected, repeats the coupling that forced this revision. Hard-code a single generic RFC 8693 client — rejected, it cannot cover header-injection/API-key fronting or per-IdP validation quirks. Dynamically-loadable third-party plugins — rejected, unacceptable attack surface for a service holding many users' credentials.
- **Update (rev 0.8.0).** ADR-017 makes the core fully provider-neutral. The **generic RFC 8693 token-exchange plugin is the reference baseline** that every other plugin conforms to; **Okta, Keycloak, Microsoft Entra (OBO), and GitHub OAuth are peer, first-class in-tree plugins** — none shipped "first" or privileged in the core, the prose, or the configuration. The provider set named in this ADR is illustrative, not ranked; the SPI stays open for additional in-tree, security-reviewed providers.

### ADR-010: Implementation language — Go
- **Decision.** Implement `obo-broker` in **Go**. This supersedes the earlier TypeScript/Node choice (a holdover from the MSAL-era reasoning in ADR-008).
- **Context.** With standards-based OIDC/OAuth2 providers (ADR-009; the generic RFC 8693 baseline plus Okta, Keycloak, Entra — ADR-017) the MSAL-only constraint is gone, reopening the language choice. The service is a small, security-critical, stateless token-exchange + proxy on self-managed Kubernetes, single-tenant per enterprise — so the decision is driven by operational fit, supply-chain surface, library maturity, and auditability, not raw compute.
- **Consequences.** Tiny static binary, fast cold start, low memory → cheap to run many single-tenant instances; memory-safe (no meaningful gap to Rust on an I/O-bound service); a lean dependency surface versus npm (material for a credential-handling service); mature standards-based OIDC/OAuth2/JWT libraries (`coreos/go-oidc`, `x/oauth2`); the Provider Plugin SPI (ADR-009) maps cleanly to Go interfaces, compiled in. Cost: diverges from the caller toolchain (now the Rust daemon) — acceptable because the broker is a separately operated deployable, so a shared toolchain is a weak benefit; the team needs Go competence.
- **Alternatives considered.** *.NET (C#)* — equally secure, first-class OIDC/JWT, heavier container images; the right choice only if the organisation standardises on .NET. *TypeScript/Node* — keeps one toolchain with the daemon but carries the largest npm supply-chain surface for a credential service; rejected as primary. *Rust* — maximal assurance but over-investment for an I/O-bound orchestration service; revisit only with existing Rust depth.

### ADR-011: Envelope-encrypt the token cache via a key-management service
- **Decision.** Encrypt cached downstream credentials with AES-256-GCM under a **per-entry data key** obtained from a key-management service (cloud KMS, or a Vault transit engine in-cluster); store the wrapped data key alongside the entry; rotate the KMS master key on a schedule. This supersedes the earlier single static service-held key. Resolves OQ-1/OQ-2 (key custody).
- **Context.** The service caches many users' privileged downstream credentials; a single static encryption key means one key disclosure decrypts every cached entry.
- **Consequences.** The blast radius of a key compromise is bounded; master-key rotation is near-zero-effort (data keys are re-wrapped, entries are short-lived and self-expire); the cache adapter calls KMS on store/decrypt. Adds a KMS dependency — consistent with the workload-identity custody already chosen for the client certificate (ADR-006). A 96-bit random nonce is used per seal and entries-per-data-key are bounded.
- **Alternatives considered.** Single static service-held key — rejected: unbounded blast radius and no rotation. Per-tenant key only — rejected: too coarse for a multi-user single-tenant instance; per-entry data keys subsume it.

### ADR-012: Provider-plugin trust-boundary guardrails
- **Decision.** Add lightweight technical controls around the plugin SPI: (1) the **core** re-validates the `Principal` a plugin returns against the expected `iss`/`aud` rather than solely trusting the plugin; (2) plugins acquire credentials through a constrained acquisition path rather than receiving raw confidential-client key material — required wherever the provider exposes a constrained-acquisition mechanism, with any plugin that genuinely cannot documenting the exception and undergoing dedicated security review (settled per-plugin in `/spec`); (3) plugins remain compiled-in, first-party, and security-reviewed (ADR-009). No per-plugin process sandboxing.
- **Context.** The SPI hands a plugin the inbound token and the credential-acquisition path; a buggy plugin could mis-validate identity or leak the credential. "First-party and reviewed" is a process control only — these add a thin technical backstop.
- **Consequences.** Defence-in-depth against a plugin bug at low cost; the core owns the final identity check. Full isolation is deemed over-engineering for in-tree code; matters more as additional plugins are added.
- **Alternatives considered.** Trust plugins entirely — rejected: a single plugin bug becomes a credential leak. Sandbox each plugin in a subprocess — rejected: disproportionate for compiled-in first-party code.

### ADR-013: Okta fronting — default to token exchange, verify per deployment
- **Decision.** Default the Okta provider plugin to **RFC 8693 token exchange against an Okta custom authorization server (API Access Management)**. Before production, verify the Okta org supports it and that the corporate APIs accept Okta-issued access tokens. If the org instead fronts apps with **Okta Access Gateway** (header injection), build the `okta-access-gateway` plugin variant that ADR-009 anticipates. Resolves OQ-7.
- **Context.** The token-exchange flow is assumed, but Okta editions/configurations differ and the fronting model materially changes the provider plugin's implementation.
- **Consequences.** Token-exchange is the portable, standards-based default; the plugin architecture contains the alternative as a swap. The Okta plugin spec (C6) carries this assumption until the org check passes.
- **Alternatives considered.** Hard-commit to access-gateway — rejected: less portable and not confirmed. Leave unspecified — rejected: the plugin spec needs a concrete default to build against.
- **Update (rev 0.8.0).** Superseded in part by ADR-017: there is **no Okta default**. The token-exchange-vs-access-gateway *fronting-flavour* verification this ADR describes still applies, but to **whichever provider a deployment configures** (Okta, Keycloak, Entra, …), with the **generic RFC 8693 token-exchange flow as the reference baseline**. The `okta-access-gateway` variant remains the worked example of a header-injection fronting plugin; an equivalent variant is built for any provider fronted that way.

### ADR-014: Step-up authentication propagates via the `id_token` `acr` claim, server-enforced, with an RFC 9470 challenge
- **Decision.** When a capability sets `requireStepUpAuth`, the broker requires the validated user `id_token` to carry an `acr` (authentication context class reference) claim whose value is in a configured allowlist (`OBO_STEP_UP_ACR_VALUES`); `amr` corroborates. On insufficiency the broker returns `401` with `WWW-Authenticate: Bearer error="insufficient_user_authentication", acr_values="<required>"` (RFC 9470), and the daemon performs an IdP step-up and retries. The step-up signal is **never** a caller-supplied request field. Resolves OQ-4 (and the parent `../sandbox-daemon/` OQ-1 propagation question).
- **Context.** The parent daemon can require step-up/MFA for sensitive (write) capabilities, but the backend is a separate trust domain and the calling agent is untrusted. The server must learn that step-up occurred without trusting a forgeable signal. OIDC already conveys assurance in the `acr`/`amr` claims, and RFC 9470 standardises the "insufficient authentication" challenge — reusing them avoids a bespoke protocol and a new credential.
- **Consequences.** Sensitive capabilities are gated end-to-end with no caller-forgeable bypass; the check rides on the already-validated `id_token` (ADR-004) and is re-validated by the core (ADR-012); the provider-agnostic PolicyEnforcer owns the comparison while the plugin only extracts the claims. Config fails closed if a capability requires step-up but no `acr` values are configured. Cost: the concrete `acr` value is Okta-org-specific and must be verified per deployment (consistent with ADR-013); the daemon must implement step-up-on-challenge and retry.
- **Alternatives considered.** A caller-supplied `stepUpSatisfied` flag on the request — rejected: trivially forgeable by the untrusted agent, defeating the control. A separate signed step-up assertion alongside the `id_token` — rejected: redundant given the `id_token` already carries `acr`/`amr`, and it adds a second validation path. Leaving the existing "claim name from config" wording — rejected: too vague to build or test against.

### ADR-015: Step-up requires recent authentication — `auth_time` freshness window (amends ADR-014, closes SR-26)
- **Decision.** For a `requireStepUpAuth` capability, the broker additionally requires the validated `id_token`'s `auth_time` claim to be within a configured recency window (`OBO_STEP_UP_MAX_AGE_SECONDS`, default proposed at 300 s, confirmed in `/spec`) of the request time — not merely that `acr` is in `OBO_STEP_UP_ACR_VALUES`. If `auth_time` is missing or older than the window, the broker returns the same RFC 9470 `401` `insufficient_user_authentication` challenge (adding `max_age` to the challenge parameters), and the daemon steps up and retries (ADR-015 parent side, `../sandbox-daemon/` ADR-015). Config fails closed if a capability requires step-up but no max-age is configured.
- **Context.** ADR-014 checks *that* the user once reached an elevated `acr`, but not *when*. Because a fresh `id_token` can retain an elevated `acr` long after the MFA that produced it, an MFA performed at 09:00 would otherwise satisfy every sensitive write for the rest of the day — defeating the "re-confirm presence for this sensitive action" intent of step-up. RFC 9470 / OIDC `max_age` exist precisely to bound authentication recency, so reusing them avoids a bespoke freshness signal.
- **Consequences.** Step-up means *recent* step-up: a sensitive write requires an MFA within the window, not merely a historical one. The check rides on the already-validated `id_token` (ADR-004/ADR-012) and the provider-agnostic PolicyEnforcer owns the comparison. Cost: users re-confirm for sensitive writes that fall outside the window — the intended user-presence cost of step-up; the window length is a per-deployment trade-off.
- **Alternatives considered.** `acr`-only (the ADR-014 wording) — rejected: assurance level without recency lets one MFA cover an unbounded span of sensitive writes. A broker-side "last step-up at" cache keyed by user — rejected: it reintroduces server state the stateless design avoids and is weaker than the `id_token`'s own signed `auth_time`.

### ADR-016: Downstream-token lifetime ceiling, revocation posture, and an admin invalidation endpoint (closes SR-29)
- **Decision.** (1) Cached downstream tokens carry a **hard maximum TTL ceiling** (`OBO_CACHE_MAX_TTL_SECONDS`, confirmed in `/spec`) in addition to `TTL ≤ token expiry` — whichever is smaller wins — so a long-lived downstream token cannot sit cached for its full natural life. (2) **Silent `refresh` (ADR/sequence) does not extend access for a deprovisioned user:** refresh is attempted only within the entry's window and a refresh that the IdP rejects (revoked grant, disabled user) evicts the entry and fails the call closed (`502 auth_refresh_failed`) — refresh is not a mechanism to outlive IdP-side revocation. (3) A protected **admin invalidation endpoint** `POST /v1/admin/invalidate` evicts cache entries by `user` (and optionally `audience`/`providerId`), so an operator can cut a user's cached downstream access immediately on deprovisioning without waiting for natural expiry. The endpoint authenticates the caller as an operator (cluster-internal, mTLS or a dedicated admin credential — not the user `id_token`) and is itself audited.
- **Context.** The cache TTL was bounded only by the downstream token's own expiry, which the service does not control and which can be long; and the domain model mentioned eviction "on explicit invalidation" without any interface exposing it. Combined, a user disabled in Okta could retain downstream access via cached tokens (and silent refresh could extend it) until natural expiry — a revocation-lag gap for a high-value service holding many users' privileged access.
- **Consequences.** Revocation lag is bounded by the smaller of the TTL ceiling and the natural expiry, and an operator has an immediate kill switch per user. Refresh can no longer extend a revoked user's access. Cost: a new authenticated admin surface (kept cluster-internal, operator-authenticated, audited) and a small cache-hit-rate reduction from the TTL ceiling — acceptable for a credential-custody service.
- **Alternatives considered.** Rely on natural token expiry alone — rejected: leaves revocation lag unbounded and offers no operator control. A pub/sub revocation feed from the IdP — viable and stronger, but heavier and IdP-specific; recorded as a `/spec`/future enhancement, with the TTL ceiling + admin endpoint as the portable baseline.

### ADR-017: Provider-neutral baseline — no privileged default IdP (amends ADR-013, extends ADR-009)
- **Decision.** The broker has **no default identity provider**. The **generic RFC 8693 token-exchange plugin is the reference implementation** and the baseline contract every other plugin conforms to; **Okta, Keycloak, Microsoft Entra (OBO), and GitHub OAuth are peer, first-class in-tree plugins**, none privileged in the core, the prose, the spec defaults, or the configuration. The active provider for each capability is selected solely by the capability's `provider` field (ADR-009) and per-deployment configuration; an unset or unknown provider fails closed at load. This supersedes ADR-013's "default to Okta token exchange" framing — Okta becomes one configured provider among peers, and the fronting-flavour check ADR-013 describes (token exchange vs header-injection access gateway) applies to whichever provider a deployment selects, not to Okta specifically.
- **Context.** Earlier revisions named Okta as "the provider for this deployment" throughout the HLD and spec even though ADR-009 had already made the core provider-agnostic, leaving the prose and defaults Okta-heavy while the architecture was not. A separate evaluation (recorded in [10 — Risks](./10-risks.md) Alternatives and ADR-008's rev-0.8.0 update) confirmed that no off-the-shelf product — including the Okta portfolio (API Access Management, Cross-App Access, Auth0 Token Vault, Okta Access Gateway) — substitutes for the broker, because each issues a token to the caller or does not proxy-and-sanitize, violating the never-return-token contract (ADR-007). The broker is therefore built and kept, but it must not privilege any single IdP.
- **Consequences.** A deployment selects Okta, Keycloak, Entra, or a generic RFC 8693 authorization server by configuration alone — no code or core change; each plugin remains a separately security-reviewed unit (ADR-009/ADR-012); the reference RFC 8693 plugin gives a vendor-neutral conformance target for the SPI; documentation, spec defaults, and tests no longer assume Okta. Cost: per-provider validation quirks (issuer/JWKS shapes, token-exchange parameter dialects, fronting models, `acr`/`amr` value sets) are verified per deployment for the chosen provider rather than once for Okta — the ADR-013/ADR-014 per-deployment verification generalises across providers.
- **Alternatives considered.** *Keep Okta as the documented reference example* — rejected this pass: even a "reference only" default leaks into spec defaults and tests and re-creates the coupling. *Pick a different single default (e.g. Keycloak)* — rejected: trades one privileged IdP for another. *Generic RFC 8693 only, no per-IdP plugins* — already rejected by ADR-009: it cannot cover header-injection/API-key fronting or per-IdP validation quirks.
- **Keycloak role disambiguation.** Keycloak appears in ADR-008/10-risks as a *rejected* option — that rejection is of **Keycloak as a replacement for the broker** (operating a full IdP as the exchange product). It does **not** reject **Keycloak as an identity provider the broker performs RFC 8693 token exchange against**, which this ADR and ADR-009 explicitly support via a `keycloak` plugin. Both stances stand.

### ADR-018: Pluggable key-management via a KeyManager SPI (extends ADR-011)
- **Decision.** Abstract the key-management dependency that ADR-011 introduced behind a **`KeyManager` SPI** — a compiled-in, first-party, security-reviewed interface with two operations, `GenerateDataKey` (mint a per-entry data key, returning the plaintext DEK to seal with and the wrapped DEK to persist) and `Unwrap` (recover the plaintext DEK from its wrapped form). The active backend is selected **at startup** by `OBO_KMS_PROVIDER` (deployment-wide, not per-capability — one KMS per single-tenant instance, ADR-002); the master-key reference stays `OBO_CACHE_ENC_KEY_REF`. Ship in-tree backends for **cloud KMS (AWS KMS / GCP KMS / Azure Key Vault) and a Vault transit engine**; there is **no default** — an unset or unknown `OBO_KMS_PROVIDER` fails closed at startup. The `TokenCacheAdapter` (C4) depends on the `KeyManager` interface, never on a concrete KMS. This mirrors the Provider Plugin model (ADR-009) and inherits its trust-boundary stance (ADR-012): backends are compiled-in, first-party, and security-reviewed — **never dynamically loaded** (a KeyManager handles raw key material, so the no-dynamic-load prohibition is even more load-bearing here than for identity plugins).
- **Context.** ADR-011 settled *that* the cache is envelope-encrypted under a KMS-derived per-entry data key, and named "cloud KMS, or a Vault transit engine in-cluster" as the options — but left no abstraction: the `TokenCacheAdapter` called "KMS" directly, coupling the credential cache to one backend. Deployments differ exactly here — a cloud deployment uses its provider's KMS; a self-managed-Kubernetes deployment (ADR-005) more naturally uses an in-cluster Vault transit engine. Key custody should be as deployment-agnostic as identity now is (ADR-017), and through the same plugin discipline rather than a second bespoke mechanism.
- **Consequences.** A deployment selects its KMS backend by configuration with no core or cache-adapter change; each backend is a separately security-reviewed unit; envelope encryption, per-entry data keys, master-key rotation, and the bounded blast radius (ADR-011) are unchanged — the SPI only relocates the wrap/unwrap calls behind an interface. The cache-decrypt failure mode (`CACHE_DECRYPT_FAIL`) and the KMS-unavailable readiness gate generalise across backends. Cost: a small versioned SPI to maintain and a per-backend integration test (the cloud backends are tested against the provider's emulator or a recorded contract; Vault transit against a dev container).
- **Alternatives considered.** Hard-code a single KMS — rejected: couples a credential-custody service to one platform and contradicts the deployment-agnostic posture (ADR-005/ADR-017). A single generic "KMS URL" config with no interface — rejected: cannot cover per-backend authentication (workload identity vs Vault token), wrap formats, and rotation semantics. Make the KMS selectable **per capability** like identity providers (ADR-009) — rejected: key custody is a deployment-wide concern with one active KMS per single-tenant instance (ADR-002), so selection belongs at startup, not in the per-request path. Dynamically-loadable KMS plugins — rejected for the same reason as ADR-009, and more strongly: the backend holds the cache's master-key path.
