# 09 — Architectural Decisions

> **Revision:** 0.3.0

Append-only. Decisions are amended via new entries, not by overwriting.

> **Numbering note.** This profile continues the **faradayd** ADR register (the sandbox-daemon HLD's `09-decisions.md`), which ends at **ADR-033**. New decisions here are **ADR-034 … ADR-038**. They amend, but do not overwrite, the sandbox-daemon ADRs they reference; the two folders share one ADR number space for the daemon. The `obo-broker` HLD keeps its own separate register.

### ADR-034: Server-side single-tenant deployment — one daemon per agent, in a container (amends A-1, ADR-023, ADR-030)
- **Decision.** Support a deployment profile in which `faradayd` runs **inside a container, single-tenant, one daemon per agent**, with the agent runtime and the daemon sharing the **same container and OS UID**. The threat-model assumption **A-1** ("one daemon per OS user; no cross-user sharing") is reinterpreted for this profile as **"one daemon per agent service"** — still a single trust principal per daemon instance. Because the agent and daemon share a UID, the existing client-authentication controls (ADR-024: same-UID peer check + `0600` socket + connection token) hold **unchanged**; **no network client-authentication (mTLS / bearer) is introduced**.
- **Context.** The desktop profile (ADR-023) is per-user and assumes a human at the machine. To serve an agent deployed on a server runtime, the daemon must run server-side. The expensive parts of a server port — multi-tenant token-vault isolation and a network client-auth boundary — only arise if a daemon serves *many* principals or accepts *remote* clients. Constraining the profile to single-tenant, one-daemon-per-agent, co-located in one container removes both: there is one principal, and the client is local.
- **Consequences.** A server-deployed agent can use faraday with the local trust model intact — TB1's mitigations apply verbatim (the agent crosses TB1 exactly as a desktop client does). No multi-tenant isolation is designed or owed (07-cross-cutting: Multi-tenancy "does not apply"). **Cost / new surface:** a containerised Linux runtime path that the per-OS service installers (ADR-030/ADR-031) do not yet cover — packaging work, tracked as a dependency/risk (10-risks). The daemon remains a single point of failure for its one agent (clients fail closed, unchanged).
- **Alternatives considered.** **Multi-tenant server daemon** — rejected for this profile: requires per-tenant vault isolation and a network client-auth boundary, a much larger design; out of scope (00-overview Non-goals). **Separate agent and daemon containers with a network socket between them** — rejected: re-introduces the network client-auth boundary this decision avoids; the single-container, single-UID shape keeps ADR-024 load-bearing without change. **Reuse the per-user OS-service model unchanged** — rejected: it assumes a logged-in human user and an OS service manager, not a container lifecycle.

### ADR-035: No OBO server-side — the broker holds the downstream credential directly (scopes ADR-002; does not use ADR-005)
- **Decision.** In the server-mode profile, downstream credentials are held **by the broker process itself**, not exchanged through the server-side `obo-broker`. The `obo-broker` token-exchange path (ADR-005) is **not used** by this profile. The one invariant is preserved unchanged: the credential is applied to the outbound request by the broker and **never reaches the guest sandbox or the returned envelope** (ADR-002; the `broker.rs` "tokens never leave this module" property).
- **Context.** `obo-broker` exists to keep a **privileged downstream token off an untrusted workstation** and away from the **human user** who owns that workstation (ADR-005 / ADR-002: "protecting tokens from the user is explicitly not a goal" — the defended boundary is the agent/sandbox). In the server-mode profile there is **no untrusted workstation** (the daemon runs in a controlled container) and **no human user** to defend against — the broker process is itself the trust anchor. OBO's rationale therefore does not apply, and the credential it would protect (the static API key) *is* the credential; there is no more-privileged token behind it to keep server-side.
- **Consequences.** A server agent makes authenticated calls with no dependency on the `obo-broker` availability and no token-exchange round-trip. The credential-custody guarantee that matters here — keeping the key out of the *guest* — is unchanged and is exactly what ADR-002 / `Passthrough` already enforce. **Cost / accepted residual:** the broker process now holds a long-lived static key in memory; a compromise of the daemon container exposes that key (bounded to the key's allowlisted host/path/method scope and budgets). This is an accepted posture for a single-tenant controlled server deployment (07-cross-cutting NFRs), not the desktop threat model.
- **Alternatives considered.** **Keep OBO server-side via workload identity** (make the daemon a confidential client / use a K8s service-account or SPIFFE identity, then RFC 8693 exchange) — rejected for this profile: it adds a confidential-client and an exchange dependency to protect against a threat (untrusted host, human user) that does not exist single-tenant; recorded as the path to revisit if a multi-tenant or delegated-user server profile is ever required. **Pass an incoming end-user assertion through to OBO** — rejected here: there is no end-user in a headless server agent; this is the natural fit only for an on-behalf-of-user server profile, which is out of scope.

### ADR-036: `api_key` auth mode — per-capability static key, file-backed, with configurable placement (extends the C11 auth-mode routing)
- **Decision.** Add an `api_key` downstream auth mode. The broker holds **one static key per capability**, resolved from a **file** via the existing `SecretResolver` (`FileSecretResolver`, the `*_REF` → file convention), named by a per-capability **`secretRef`**. The policy author declares **how** the key is attached to the request — a named request header with a scheme/prefix, or a query parameter — via a per-capability **key-placement** descriptor; the mode does **not** assume `Authorization: Bearer`. DownstreamClient (C10) applies the key at that placement; the broker never serialises it into the response.
- **Context.** A server agent needs to call API-keyed third-party services. The existing modes both require a human OIDC sign-in (00-overview Motivation), which a headless server agent cannot perform. A static key is the standard credential such APIs accept. Two sub-decisions follow: **where the key comes from** and **how it is attached**. The repo already resolves secrets from files (`FileSecretResolver`), and that mechanism is safer than process environment variables (which leak via `/proc`, crash dumps, and child processes). Real-world keyed APIs use varied schemes (`Authorization: Token`, `X-API-Key`, query params), so a Bearer-only mode would not serve them.
- **Consequences.** A keyed API becomes reachable by a headless agent with no sign-in and no OBO. The key custody guarantee mirrors `Passthrough` exactly — applied by the broker, never exposed to the guest (ADR-035). **Cost:** the policy schema gains `secretRef` and a key-placement descriptor (08-interfaces; wire shape in `/spec`); config-load validates them. The static-key blast-radius and rotation concerns are recorded as posture (07-cross-cutting NFRs) and risk (10-risks).
- **Alternatives considered.** **Raw environment-variable keys** (the literal "API_KEY as env var" framing) — rejected: env vars leak more readily than a `0600` mounted file, and the repo already standardises on `*_REF` → file; a mounted Kubernetes/Docker secret referenced by `secretRef` is both safer and idiomatic. **Bearer-only `api_key`** — rejected: many keyed APIs do not use Bearer; a fixed scheme would exclude them, so placement is configurable. **A single daemon-level key env var** — rejected: keys are per-resource, so they belong on the capability, mirroring the ADR-033 reasoning that audiences belong on the capability, not on one global setting.

### ADR-037: `none` auth mode — unauthenticated calls to public endpoints, still allowlist-bounded (extends the C11 auth-mode routing)
- **Decision.** Add a `none` downstream auth mode for genuinely public endpoints. The broker sends **no credential** and triggers **no sign-in**. A `none` capability is **still** fully constrained by the policy allowlist (host / path / method), path canonicalisation, per-run/session budgets, and audit — `none` means "no credential", not "any call". `provider` and `audience` are not required for a `none` capability.
- **Context.** Public datasets (e.g. `https://www.gov.uk/bank-holidays.json`) require no credential, but today every outbound call must resolve to a capability whose auth mode attaches a token; there is no way to make a credential-free call, so an unmatched call fails closed (`CAP_UNKNOWN`). A server agent that needs public data therefore has no path. Adding `none` keeps the fail-closed posture (the author must still allowlist the host/path/method) while removing the false requirement of a credential.
- **Consequences.** Public APIs become reachable, and a run composed only of `none` (and/or `api_key`) capabilities needs no human and no IdP. The security boundary is unchanged because the allowlist, budgets, and audit still apply; the only relaxation is "no `Authorization` header is attached". **Cost:** one more `authMode` value and a broker arm that applies no credential (02-architecture). A `none` capability still returns the untrusted-content envelope, so the prompt-injection stance (ADR-017) is unchanged.
- **Alternatives considered.** **Reuse `passthrough` with an empty token** — rejected: it would still drive the sign-in/audience machinery and conflate "no credential" with "the user's credential"; a distinct mode is clearer and avoids requiring an IdP. **Leave public APIs unsupported and require a dummy credential** — rejected: it forces meaningless configuration and an IdP dependency for a credential-free call.

### ADR-038: OIDC configuration is optional when no OIDC-backed capability is present (relaxes the C1 mandatory-OIDC rule)
- **Decision.** The OIDC configuration group (`PYS_OIDC_ISSUER` and `PYS_OIDC_CLIENT_ID`, currently both required at config load) becomes **conditional**: required only when the resolved policy manifest contains at least one OIDC-backed capability (`exchange` or `passthrough`); **not required** when every capability is `api_key` or `none`. All other startup invariants — fail-closed on missing required values, and the ADR-016 rule that real credentials require a reachable OTLP audit sink — are **unchanged**.
- **Context.** A pure server-mode deployment uses no OIDC at all, yet the daemon today refuses to start without an OIDC issuer and client id (`config.rs` marks both `required`). That makes a credential-free / key-only deployment impossible to start. The manifest already declares each capability's auth mode, so the daemon can determine at load time whether OIDC is needed.
- **Consequences.** A key-only / public-only deployment starts with no OIDC configuration and binds no loopback sign-in listener. A mixed deployment (some OIDC-backed capabilities present) still requires the OIDC group, preserving the desktop behaviour. **Cost:** config validation gains a manifest-dependent conditional (the OIDC group's requiredness depends on the parsed capabilities); the exact validation ordering and error codes are `/spec`'s to fix. The ADR-016 audit-sink gate is deliberately **not** relaxed — a real `api_key` still implies an authoritative audit trail.
- **Alternatives considered.** **Keep OIDC mandatory and require a dummy issuer** — rejected: forces meaningless configuration and a non-functional sign-in surface in a profile that never signs in. **A separate "server-mode" flag that blanket-disables OIDC** — rejected: a flag can contradict the manifest (OIDC capabilities present but OIDC disabled); deriving the requirement from the manifest's actual capabilities is self-consistent and fail-closed.

### ADR-039: Sensitive (write) capabilities are per-capability opt-in, default read-only (resolves OQ-SM-1; amends ADR-025, builds on ADR-021)
- **Decision.** In server-mode a capability is **read-only by default**: with no explicit opt-in, a capability may declare only the safe method `GET`, and any unsafe method (`POST` / `PUT` / `PATCH` / `DELETE`) is rejected at config load. A capability may perform writes only if it carries an explicit per-capability **write opt-in flag** (conceptually `allowWrite`; exact field name fixed in `/spec`) in the policy manifest. The opt-in is honoured **only in a trusted / administrator-signed manifest** ([sandbox-daemon ADR-021](../sandbox-daemon/09-decisions.md)) — the admin-signed policy is the **pre-grant** that authorises the write, satisfying ADR-025's "pre-granted consent via signed policy" non-interactive posture in place of a human-present renderer. **Step-up** (ADR-015 / ADR-025 human re-auth) is **not applicable** in server-mode — there is no human and no `id_token` to carry the elevated `acr` — so the signed-policy pre-grant plus the explicit opt-in flag is the control that stands in for it (resolves OQ-SM-4). Per-session interactive consent (`requireUserConsentPerSession`) is therefore a **no-op** in headless server-mode: the signed policy is the authorisation of record (resolves OQ-SM-5).
- **Context.** [sandbox-daemon ADR-025](../sandbox-daemon/09-decisions.md) requires a human-present renderer for a sensitive (write) capability with real credentials, and fails it closed otherwise — but a headless server agent has no human. Three answers were possible: (a) trust the admin-signed policy alone for all writes, (b) make server-mode read-only, (c) require an explicit per-capability opt-in with a read-only default. The user chose (c): it is **default-safe** (a minimal or careless manifest can only read), while still allowing writes where an administrator has explicitly and verifiably authorised them.
- **Consequences.** A server agent can perform writes only against capabilities an administrator deliberately marked writable in a signed manifest; everything else is read-only, and an unsafe method on a non-opted-in capability fails closed at load. This makes the ADR-025 write-gate enforceable without a human: the signature is the pre-grant, the opt-in flag is the per-capability authorisation, and the existing method allowlist (enforced at the broker, SR-2 / SR-6) is the runtime check. **Cost / surface:** the policy schema gains the write opt-in flag (08-interfaces); Config (C1) gains a load-time validation rule (a non-opted-in capability may declare only `GET`); the unsigned-override rejection of ADR-021 now also governs whether writes can be enabled at all. The read-only default means an existing read-only manifest needs no change.
- **Alternatives considered.** **(a) Trust the signed policy for all writes** (no per-capability flag) — rejected: not default-safe; any capability listing an unsafe method would silently be a write grant, so a careless manifest widens authority by omission. **(b) Read-only server-mode** — rejected: too limiting; keyed third-party APIs (ADR-036) commonly require writes, and the user wants them reachable. **A daemon-level "allow writes" switch** — rejected: it is all-or-nothing and decoupled from the specific capability and its signature, exactly the global-setting anti-pattern ADR-033 and ADR-036 avoid.
- **Amendment (2026-06-19, revision 0.3.0).** The read-only-by-default write gate is enforced **globally** in `PolicyEngine::load` (C4), which makes no profile distinction — "server-mode" is a deployment shape, not a runtime mode of the daemon. The gate therefore applies to **every manifest, all profiles** (desktop included): any capability declaring an unsafe method needs `allowWrite: true`, honoured only in an admin-signed manifest. The original "in server-mode" framing is superseded by "all manifests". (Surfaced by `make ci` when the shipped demo manifest's `POST` capability required the opt-in.)

## Open questions resolved in this register

- **OQ-SM-1** (blocking) — *may headless server-mode perform write capabilities with a stored key?* **Resolved by ADR-039**: yes, but only via an explicit per-capability write opt-in in an admin-signed manifest; read-only by default.
- **OQ-SM-4** — *does `api_key` interact with step-up?* **Resolved by ADR-039**: step-up is not applicable in server-mode; the signed-policy pre-grant plus the write opt-in flag is the control.
- **OQ-SM-5** — *what does `requireUserConsentPerSession` mean with no human?* **Resolved by ADR-039**: per-session interactive consent is a no-op in headless server-mode; the admin-signed policy is the authorisation of record.
